## Per-step execution contract

Run **each stub in its own fresh Claude Code session.** Same prompt shape every time:

```
Read .aipe/project/context.md and .aipe/specs/refactors/<stub>.md.

1. Run the tests. They must pass — if they don't, stop and surface the
   pre-existing failure before touching anything.
2. Refactor exactly as the spec says. Do not change behaviour.
   Do not touch anything outside the listed scope.
3. Run the tests again. They must pass with no flake.
4. Print a one-paragraph summary of what changed, citing file:line.
```

**Commit immediately after each refactor.** One refactor per commit. Each commit must be independently bisectable, so a regression caught next week pins to exactly one cleanup item.

**Push when convenient** — batching 2–3 commits into a push is fine, but never batch refactors into one session.

## Per-stub completion checklist

The cleanup-audit spec requires each stub's `## Must not change` and `## Must not introduce` sections — currently blank in every stub — to be filled before execution. Either:

**(a)** Fill them inline before running the session (preferred — keeps the contract explicit), or
**(b)** Tell the Claude session at execution time: *"The Must-not-change section is empty; treat it as: external API identical, no behaviour change, no new dependencies, no new abstractions."*

Option (a) takes ~5 minutes per stub and makes the spec self-contained for future re-runs. Option (b) is faster but leaves the discipline implicit.

## Phase 2 — preconditions for the `app/page.tsx` refactor

The audit **deliberately punted** the `app/page.tsx` shallow-module refactor (817 LOC, now 15 `useState`) to `fix-later`. Two preconditions must land first:

### Precondition A — Extract `parseNdjsonStream` (`fix-later #24`)

The NDJSON parsing loop is currently duplicated across **four** files:

```
app/page.tsx:450-456                 (main effect)
app/page.tsx (capture path)          (demo-capture flow)
lib/hooks/useInvestigation.ts:184-208 (investigate page)
components/feed/StreamingResponse.tsx (streaming card)
```

**The refactor:** lift the loop into `lib/mcp/ndjson.ts` as an async generator:

```ts
async function* parseNdjsonStream<T>(
  reader: ReadableStreamDefaultReader<Uint8Array>
): AsyncGenerator<T>
```

Each call site becomes `for await (const event of parseNdjsonStream<AgentEvent>(res.body.getReader())) { handle(event) }`. **Pure function, no React/fetch dependencies, ~25 LOC.** Testable in isolation.

**To execute:** hand-compose a stub at `.aipe/specs/refactors/cleanup-extract-ndjson-stream.md` (the cleanup audit didn't emit one because it's `fix-later`), then run a fresh session against it. ~1 hour.

### Precondition B — Route-level integration tests (`fix-later #15`)

The repo has 169 unit tests but few end-to-end. Without integration coverage that exercises `app/page.tsx` → `/api/briefing` → agent loop, the page.tsx hooks-extraction has nothing to verify "behaviour preserved" against. The audit named this as the second precondition — same `fix-later` status, no stub yet.

**Scope:** ~5 integration tests covering the main feed-page lifecycle (mount, stream, reconnect, mode toggle, error). Estimate ~1 day.

## Hard rules (carried verbatim from the cleanup audit spec)

These are the spec's discipline gates, not stylistic preferences:

> **Never batch cleanup refactors into one session.** Even if each is small, the combined diff destroys your ability to attribute a regression. Cleanup compounds bugs faster than feature work because nothing's supposed to have changed.

> **If the refactor wants to grow past behaviour-preserving, abandon it.** Half a cleanup that changed behaviour is worse than no cleanup, because you've now made the code different without a feature reason and lost the ability to bisect.

> **No new dependencies, no new abstractions not discussed in the spec, no additional refactors discovered along the way.** Surface those as separate items for the next cleanup audit.
