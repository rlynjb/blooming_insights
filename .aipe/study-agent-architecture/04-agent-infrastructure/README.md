# 04 — Agent infrastructure

The cross-cutting disciplines that sit under every reasoning pattern (sub-section 01), under every retrieval shape (02), and under every multi-agent topology (03). These are the parts most teams underweight and the parts that separate a demo from a shipped system. blooming insights has four of the five in production form; the fifth (automated trajectory evaluation) is honestly deferred and named that way.

Each file extends a concept covered mechanically in `../../study-ai-engineering/` toward the agent-architecture lens — context engineering as the *discipline* that decides what fills the window, memory as the *tiered storage* model, tool calling as the *protocol substrate* under every pattern, evaluation as the *trajectory* (not just the output), and guardrails as the *control envelope* around the loop. Read each file for the agent-architecture angle; follow the cross-references for the per-call mechanics.

## Files

- **[01-context-engineering.md](01-context-engineering.md)** — The discipline that decides what fills the window for the next turn — and, in a multi-agent run, which agent sees what. blooming insights actively curates at three boundaries: schema summary with three caps (`MAX_EVENTS=20`, `MAX_PROPS_PER_EVENT=10`, `MAX_CPROPS=30` in `lib/agents/monitoring.ts` L21–L33), `{categories}` checklist injection via `.replace()` (`monitoring.ts` L83–L86), and a 16,000-char tool-result truncation at the loop seam (`lib/agents/base.ts` L29/L150/L171), with a 4,000-char stream-payload cap on top (`app/api/agent/route.ts` L99–L103). The discipline is "decide the slice before you decide the prompt." **Case A — built.**

- **[02-agent-memory-tiers.md](02-agent-memory-tiers.md)** — Memory as storage layering by durability: working (this run, in-context), episodic (recent, exact-key), long-term (persistent, retrieved by relevance). Working tier = the per-run `messages` array in `runAgentLoop` (`base.ts` L79). Episodic tier = server-side `Map` in `lib/state/investigations.ts` plus client-side `sessionStorage` stash + the `bi:diag:<id>` cross-step handoff in `lib/hooks/useInvestigation.ts` L18–L19, L133, L138. Long-term tier (vector / semantic) is honestly absent — no embeddings, no vector store, no cross-run recall. **Case A — partial (working + episodic).**

- **[03-tool-calling-and-mcp.md](03-tool-calling-and-mcp.md)** — The substrate every reasoning pattern stands on. blooming insights runs it through MCP: `@modelcontextprotocol/sdk`'s `StreamableHTTPClientTransport` wrapped by `SdkTransport`, authenticated via `BloomreachAuthProvider` (PKCE + DCR) in `lib/mcp/connect.ts`, dispatched through `McpClient` (`lib/mcp/client.ts`) which owns 1.1s proactive spacing, parsed-hint rate-limit retry with exponential backoff (cap 3 retries / 20s ceiling), a 60s TTL response cache, and a no-cache-on-error rule. Per-agent surfaces are a *filter* over the discovered tool list (`filterToolSchemas` in `lib/agents/tool-schemas.ts` L15) — drift is impossible because the source is one server. **Case A — built.**

- **[04-agent-evaluation.md](04-agent-evaluation.md)** — Evaluating an agent expands the unit of eval from {input → output} to {input → trajectory → output}. blooming insights has the substrate: ~169 vitest tests TDD'd with injected `McpCaller` + Anthropic fakes pin down the loop's *shape* invariants (forced-final tool removal, budget enforcement, parse-failure safe defaults, schema-gated category list), and every run produces an `AgentEvent[]` trajectory streamed live and cached server-side (`lib/state/investigations.ts`). What's honestly absent is the automated trajectory-eval harness — no frozen input set, no MCP fixtures, no trajectory scorer, no golden-diff CI. **Case A partial (loop-shape tests + inspectable trajectory) + Case B partial (no trajectory-eval harness).**

- **[05-guardrails-and-control.md](05-guardrails-and-control.md)** — The control envelope around an autonomous loop, in five composed layers: input scoping via `runnableCategories` (`lib/agents/categories.ts` L116–L160), iteration + budget caps with the forced-final tool-strip (`base.ts` L73–L101; `maxToolCalls` is 6/6/6/4 across monitoring/diagnostic/query/recommendation; Vercel `maxDuration=300`), the read-only MCP tool contract (`lib/mcp/tools.ts` L5–L40 — every entry is `list_*` / `get_*` / `execute_analytics_*`, so the LLM's output never triggers a side effect directly), output validators (`parseAgentJson` + `isAnomalyArray` / `isDiagnosis` / `isRecommendationArray` in `lib/mcp/validate.ts`), and a one-time guarded auto-reconnect on revoked tokens (`app/page.tsx` L394–L427 keyed on `sessionStorage['bi:reconnecting']`). The principle is defense-in-depth: every freedom the loop has gets a cap, every output gets a validator, every tool that *could* be writable is read-only on purpose. **Case A — built.**

## Reading order

Read in this order:

**01 → 02 → 03 → 05 → 04.**

That is: **the discipline** that decides what fills the window (01) before **where each fact lives** by tier (02), before **the substrate** that makes the tool calls actually happen (03), before **the envelope** that bounds what the loop can do with that substrate (05), and finally **how you'd know it works** — and what you can't know automatically yet (04). The first four are what the system *is*; the fifth is how you'd verify it. Reading 04 last makes the gaps it names land as honest gaps, not as missing prereqs.

A frontend engineer reading this directory should walk away with five takeaways: (1) context engineering is your `useMemo` instinct applied to the model's working set, (2) agent memory is your React-state / sessionStorage / localStorage instinct applied to an agent's knowledge, (3) MCP is the `/api` boundary for LLMs — one definition, every agent reuses it, sliced at the boundary, (4) the control envelope is defense-in-depth around a fallible-and-manipulable model, and (5) trajectory eval is what you build the day model-quality drift starts mattering more than safety drift does.

This sub-section's anchor is **single-agent + multi-agent (both)** — every concept here applies to a single ReAct loop and to a multi-agent topology. The four single-agent concepts in this codebase exercise all five disciplines today; if blooming insights ever grows a supervisor-worker shape (covered in sub-section 03), these same files describe what the supervisor and each worker would each need.
