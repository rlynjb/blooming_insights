# 04 — Agent infrastructure

The cross-cutting disciplines that sit under every reasoning pattern (sub-section 01), under every retrieval shape (02), and under every multi-agent topology (03). These are the parts most teams underweight and the parts that separate a demo from a shipped system. blooming insights has five in production form — context engineering, memory tiers, tool calling and MCP, guardrails, and the AptKit runtime layer (the senior pattern that landed when the active spine moved out of Blooming's `runAgentLoop` and into `@aptkit/core`). Agent evaluation has the substrate (structural unit tests + the inspectable `AgentEvent[]` trajectory) but not the automated harness — that pipeline was removed when `eval/` and the sibling `mcp-server-olist/` package were deleted in PR #8.

Each file extends a concept covered mechanically in `../../study-ai-engineering/` toward the agent-architecture lens — context engineering as the *discipline* that decides what fills the window, memory as the *tiered storage* model, tool calling as the *protocol substrate* under every pattern, evaluation as the *trajectory* (not just the output), and guardrails as the *control envelope* around the loop. Read each file for the agent-architecture angle; follow the cross-references for the per-call mechanics.

## Files

- **[01-context-engineering.md](01-context-engineering.md)** — The discipline that decides what fills the window for the next turn — and, in a multi-agent run, which agent sees what. blooming insights actively curates at three boundaries: schema summary with three caps (`MAX_EVENTS=20`, `MAX_PROPS_PER_EVENT=10`, `MAX_CPROPS=30` in `lib/agents/monitoring.ts` L21–L33), `{categories}` checklist injection via `.replace()` (`monitoring.ts` L83–L86), and a 16,000-char tool-result truncation at the loop seam (`lib/agents/base.ts` L29/L150/L171), with a 4,000-char stream-payload cap on top (`app/api/agent/route.ts` L99–L103). The discipline is "decide the slice before you decide the prompt." **Case A — built.**

- **[02-agent-memory-tiers.md](02-agent-memory-tiers.md)** — Memory as storage layering by durability: working (this run, in-context), episodic (recent, exact-key), long-term (persistent, retrieved by relevance). Working tier = the AptKit agent's internal per-run conversation (active path; the legacy `messages` array in `base-legacy.ts` is preserved but inactive). Episodic tier = server-side `Map` in `lib/state/investigations.ts` plus client-side `sessionStorage` stash + the `bi:diag:<id>` cross-step handoff in `lib/hooks/useInvestigation.ts`. Long-term tier (vector / semantic) is honestly absent — no embeddings, no vector store, no cross-run recall. **Case A — partial (working + episodic).**

- **[03-tool-calling-and-mcp.md](03-tool-calling-and-mcp.md)** — The substrate every reasoning pattern stands on. blooming insights runs it through MCP: `@modelcontextprotocol/sdk`'s `StreamableHTTPClientTransport` wrapped by `SdkTransport`, authenticated via `BloomreachAuthProvider` (PKCE + DCR) in `lib/mcp/connect.ts`, dispatched through `BloomreachDataSource` (`lib/data-source/bloomreach-data-source.ts`) which owns 1.1s proactive spacing, parsed-hint rate-limit retry with exponential backoff (cap 3 retries / 20s ceiling), a 60s TTL response cache, and a no-cache-on-error rule. Per-agent surfaces are a *filter* over the discovered tool list (`filterToolSchemas` in `lib/agents/tool-schemas.ts`) plumbed through the AptKit `ToolRegistry` adapter — drift is impossible because the source is one server. The `DataSource` seam above MCP carries two adapters now: `BloomreachDataSource` and `SyntheticDataSource` (in-process). **Case A — built.**

- **[04-agent-evaluation.md](04-agent-evaluation.md)** — Evaluating an agent expands the unit of eval from {input → output} to {input → trajectory → output}. blooming insights has the substrate — 221 vitest tests under `test/` pin down the loop's *shape* invariants with injected `McpCaller`/`DataSource` + Anthropic fakes; every run produces an `AgentEvent[]` trajectory. It does **not** have an automated trajectory-eval harness in the repo today — that pipeline (the four-pillar eval suite under `eval/` and the seeded `mcp-server-olist/` subprocess it ran against) was removed in PR #8 commit 62c24d7. The honest current measurement story is structural tests + spot-checks against the inspectable trajectory. **Case A — partial (substrate present, harness absent).**

- **[05-guardrails-and-control.md](05-guardrails-and-control.md)** — The control envelope around an autonomous loop, in five composed layers: input scoping via `runnableCategories` (`lib/agents/categories.ts`), iteration + budget caps with the forced-final tool-strip (now owned by AptKit's agent classes; the legacy Blooming-side `maxToolCalls` 6/6/6/4 is preserved in `*-legacy.ts`), Vercel `maxDuration=300`, the read-only MCP tool contract (`lib/mcp/tools.ts` — every entry is `list_*` / `get_*` / `execute_analytics_*`, so the LLM's output never triggers a side effect directly), output validators (`parseAgentJson` + `isAnomalyArray` / `isDiagnosis` / `isRecommendationArray` in `lib/mcp/validate.ts`), and a one-time guarded auto-reconnect on revoked tokens (`app/page.tsx` keyed on `sessionStorage['bi:reconnecting']`). The principle is defense-in-depth: every freedom the loop has gets a cap, every output gets a validator, every tool that *could* be writable is read-only on purpose. **Case A — built.**

- **[06-aptkit-runtime-layer.md](06-aptkit-runtime-layer.md)** — The senior AI-engineering pattern that landed when the active ReAct spine moved out of Blooming's `runAgentLoop` and into `@aptkit/core` v0.3.0: *own your domain (the per-agent wrappers, the bridge adapters, the data plane); defer reusable agent behavior to a library.* Blooming owns three adapter classes in `lib/agents/aptkit-adapters.ts` (206 LOC) bridging two reusable libraries (the Anthropic SDK on one side, AptKit on the other) to its existing seams (`DataSource`, NDJSON trace hooks). The legacy Blooming spine is preserved at `base-legacy.ts` for revertibility. **Case A — built.**

## Reading order

Read in this order:

**01 → 02 → 03 → 06 → 05 → 04.**

That is: **the discipline** that decides what fills the window (01) before **where each fact lives** by tier (02), before **the substrate** that makes the tool calls actually happen (03), before **the runtime layer** that orchestrates the loop above that substrate (06), before **the envelope** that bounds what the loop can do (05), and finally **how you know it works** — honestly named, including what's absent (04). The first five are what the system *is*; the sixth is how it's verified.

A frontend engineer reading this directory should walk away with six takeaways: (1) context engineering is your `useMemo` instinct applied to the model's working set, (2) agent memory is your React-state / sessionStorage / localStorage instinct applied to an agent's knowledge, (3) MCP is the `/api` boundary for LLMs — one definition, every agent reuses it, sliced at the boundary — and the `DataSource` seam above MCP is what makes the protocol swappable for a synthetic in-process backend, (4) the AptKit runtime layer is the "own your domain + adapter layer, defer reusable agent behavior to a library" pattern as concrete code — three bridge adapters, four thin wrappers, the legacy spine preserved, (5) the control envelope is defense-in-depth around a fallible-and-manipulable model, and (6) eval at agent scale begins with structural tests + the inspectable trajectory and earns automated-harness status only when the harness ships — which it doesn't, today.

This sub-section's anchor is **single-agent + multi-agent (both)** — every concept here applies to a single ReAct loop and to a multi-agent topology. The four single-agent concepts in this codebase exercise all six disciplines today; if blooming insights ever grows a supervisor-worker shape (covered in sub-section 03), these same files describe what the supervisor and each worker would each need.
