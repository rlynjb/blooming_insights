# 04 ‚Äî Agent infrastructure

The cross-cutting disciplines that sit under every reasoning pattern (sub-section 01), under every retrieval shape (02), and under every multi-agent topology (03). These are the parts most teams underweight and the parts that separate a demo from a shipped system. blooming insights has all five in production form ‚Äî context engineering, memory tiers, tool calling and MCP, agent evaluation (the Phase 3 four-pillar suite), and guardrails. Phase 2 swapped a `DataSource` seam under tool calling (the legacy Bloomreach MCP client is now one adapter of two); Phase 3 built the eval harness from scratch on the seeded `mcp-server-olist` subprocess.

Each file extends a concept covered mechanically in `../../study-ai-engineering/` toward the agent-architecture lens ‚Äî context engineering as the *discipline* that decides what fills the window, memory as the *tiered storage* model, tool calling as the *protocol substrate* under every pattern, evaluation as the *trajectory* (not just the output), and guardrails as the *control envelope* around the loop. Read each file for the agent-architecture angle; follow the cross-references for the per-call mechanics.

## Files

- **[01-context-engineering.md](01-context-engineering.md)** ‚Äî The discipline that decides what fills the window for the next turn ‚Äî and, in a multi-agent run, which agent sees what. blooming insights actively curates at three boundaries: schema summary with three caps (`MAX_EVENTS=20`, `MAX_PROPS_PER_EVENT=10`, `MAX_CPROPS=30` in `lib/agents/monitoring.ts` L21‚ÄìL33), `{categories}` checklist injection via `.replace()` (`monitoring.ts` L83‚ÄìL86), and a 16,000-char tool-result truncation at the loop seam (`lib/agents/base.ts` L29/L150/L171), with a 4,000-char stream-payload cap on top (`app/api/agent/route.ts` L99‚ÄìL103). The discipline is "decide the slice before you decide the prompt." **Case A ‚Äî built.**

- **[02-agent-memory-tiers.md](02-agent-memory-tiers.md)** ‚Äî Memory as storage layering by durability: working (this run, in-context), episodic (recent, exact-key), long-term (persistent, retrieved by relevance). Working tier = the per-run `messages` array in `runAgentLoop` (`base.ts` L79). Episodic tier = server-side `Map` in `lib/state/investigations.ts` plus client-side `sessionStorage` stash + the `bi:diag:<id>` cross-step handoff in `lib/hooks/useInvestigation.ts` L18‚ÄìL19, L133, L138. Long-term tier (vector / semantic) is honestly absent ‚Äî no embeddings, no vector store, no cross-run recall. **Case A ‚Äî partial (working + episodic).**

- **[03-tool-calling-and-mcp.md](03-tool-calling-and-mcp.md)** ‚Äî The substrate every reasoning pattern stands on. blooming insights runs it through MCP: `@modelcontextprotocol/sdk`'s `StreamableHTTPClientTransport` wrapped by `SdkTransport`, authenticated via `BloomreachAuthProvider` (PKCE + DCR) in `lib/mcp/connect.ts`, dispatched through `McpClient` (`lib/mcp/client.ts`) which owns 1.1s proactive spacing, parsed-hint rate-limit retry with exponential backoff (cap 3 retries / 20s ceiling), a 60s TTL response cache, and a no-cache-on-error rule. Per-agent surfaces are a *filter* over the discovered tool list (`filterToolSchemas` in `lib/agents/tool-schemas.ts` L15) ‚Äî drift is impossible because the source is one server. **Case A ‚Äî built.**

- **[04-agent-evaluation.md](04-agent-evaluation.md)** ‚Äî Evaluating an agent expands the unit of eval from {input ‚Üí output} to {input ‚Üí trajectory ‚Üí output}. blooming insights has the substrate (269 vitest tests across `test/` + `mcp-server-olist/test/` pin down the loop's *shape* invariants with injected `McpCaller`/`DataSource` + Anthropic fakes, every run produces an `AgentEvent[]` trajectory) **and** the Phase 3 four-pillar suite under `eval/`: detection (deterministic 3-criterion matcher, K=10, **37% LOOSE precision / 33.3% recall** vs three seeded Olist anomalies), diagnosis (Sonnet-as-judge over 5-criterion rubric, **53.3% pass**), recommendation (Sonnet-as-judge over 3-criterion rubric, **100% pass**), and regression (capture + score against 10 golden fixtures, **30% baseline**). Rubric prompts live versioned at `eval/judges/*.md`. The eval flywheel (PR D ‚Üí Phase 2.5 fix ‚Üí PR E BRL bug ‚Üí PR F recurrence catch ‚Üí PR G drift baseline) is what makes the numbers earn their place. **Case A ‚Äî built.**

- **[05-guardrails-and-control.md](05-guardrails-and-control.md)** ‚Äî The control envelope around an autonomous loop, in five composed layers: input scoping via `runnableCategories` (`lib/agents/categories.ts` L116‚ÄìL160), iteration + budget caps with the forced-final tool-strip (`base.ts` L73‚ÄìL101; `maxToolCalls` is 6/6/6/4 across monitoring/diagnostic/query/recommendation; Vercel `maxDuration=300`), the read-only MCP tool contract (`lib/mcp/tools.ts` L5‚ÄìL40 ‚Äî every entry is `list_*` / `get_*` / `execute_analytics_*`, so the LLM's output never triggers a side effect directly), output validators (`parseAgentJson` + `isAnomalyArray` / `isDiagnosis` / `isRecommendationArray` in `lib/mcp/validate.ts`), and a one-time guarded auto-reconnect on revoked tokens (`app/page.tsx` L394‚ÄìL427 keyed on `sessionStorage['bi:reconnecting']`). The principle is defense-in-depth: every freedom the loop has gets a cap, every output gets a validator, every tool that *could* be writable is read-only on purpose. **Case A ‚Äî built.**

## Reading order

Read in this order:

**01 ‚Üí 02 ‚Üí 03 ‚Üí 05 ‚Üí 04.**

That is: **the discipline** that decides what fills the window (01) before **where each fact lives** by tier (02), before **the substrate** that makes the tool calls actually happen (03), before **the envelope** that bounds what the loop can do with that substrate (05), and finally **how you know it works** ‚Äî with real numbers (04). The first four are what the system *is*; the fifth is how it's verified. Reading 04 last lets the four pillars land against the four disciplines they grade.

A frontend engineer reading this directory should walk away with five takeaways: (1) context engineering is your `useMemo` instinct applied to the model's working set, (2) agent memory is your React-state / sessionStorage / localStorage instinct applied to an agent's knowledge, (3) MCP is the `/api` boundary for LLMs ‚Äî one definition, every agent reuses it, sliced at the boundary ‚Äî and the `DataSource` seam above MCP is what makes the protocol swappable for an authored local server, (4) the control envelope is defense-in-depth around a fallible-and-manipulable model, and (5) eval at agent scale is four-pillared (deterministic detection ¬∑ rubric diagnosis ¬∑ rubric recommendation ¬∑ capture-and-score regression), with the flywheel ‚Äî measurement ‚Üí fix ‚Üí re-measurement ‚Üí next discovery ‚Äî as the actual win.

This sub-section's anchor is **single-agent + multi-agent (both)** ‚Äî every concept here applies to a single ReAct loop and to a multi-agent topology. The four single-agent concepts in this codebase exercise all five disciplines today; if blooming insights ever grows a supervisor-worker shape (covered in sub-section 03), these same files describe what the supervisor and each worker would each need.
