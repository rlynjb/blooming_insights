# Plan-and-execute

*Industry name: plan-and-execute · Language-agnostic*

## Zoom out

```
  Zoom out — plan-and-execute is a phase split on top of the kernel

  ┌─ agent loop skeleton ────────────────────────┐
  │  step + execute + accumulate + terminate     │
  └───────────────┬──────────────────────────────┘
                  │
                  ▼
  ┌─ step() shapes ──────────────────────────────┐
  │  ReAct (default, this repo)                  │
  │  ★ plan-and-execute (structured tasks) ★     │ ← we are here
  │  reflexion, tree-of-thoughts …               │
  └──────────────────────────────────────────────┘
```

## Zoom in

Separate the *plan* from the *execution*. One expensive model call builds the whole plan up front (list of steps + dependencies); many cheap calls run each step. You decouple the strategy from the grunt work. This repo does not currently use this pattern — the diagnostic path is dynamic enough that plan-and-execute would build stale plans. This file is here so you can defend that choice and know when to reach for it.

## Structure pass

Layers: **plan phase** — **execute phase** — **re-plan trigger** (optional).

Axis to hold constant: **when does the model do the hard thinking?**

```
  Where the expensive cognition lives

  ReAct:            spread across every turn
                    (every turn is a full model call)

  Plan-and-execute: concentrated in turn 1 (the plan)
                    then every turn is a cheap execution call
```

## How it works

### Move 1 — the shape

You've written a build pipeline before: `parse → typecheck → compile → link → package`. Each step is dumb; the plan (the pipeline steps) is written once. Plan-and-execute is that shape where the plan itself is generated by the model per task, not written by the engineer.

```
  Plan-and-execute — two phases, one plan reused

  ┌─ Plan phase (expensive model, one call) ────────────┐
  │  input: task                                        │
  │  ──►  plan = [step1, step2, step3, …]                │
  │       (list of steps + optional dependencies)       │
  └──────────────────┬──────────────────────────────────┘
                     │  plan
                     ▼
  ┌─ Execute phase (cheap model, many calls) ───────────┐
  │  for step in plan:                                  │
  │    result = executor(step, prior_results)           │
  │    if diverges_from_plan(result):                   │
  │       break → re-plan                               │
  └─────────────────────────────────────────────────────┘
```

### Move 2 — what would change in this repo

**Not yet implemented.** No agent in this repo uses plan-and-execute. The reason is diagnostic: the next EQL depends on the last observation, so a plan built on turn 1 (before seeing any data) would be stale by turn 2. ReAct's per-turn re-decision fits the exploratory shape.

**Where it would earn its keep.** The RecommendationAgent is a plausible candidate. It has a more structured job: given a diagnosis, propose 1-3 concrete Bloomreach actions. A plan could look like:

```
  Hypothetical plan for RecommendationAgent

  turn 1 (plan): "For payment-failure diagnosis, propose:
                  1. campaign to notify affected users
                  2. voucher for the failed order attempts
                  3. experiment on alternate payment provider"

  turn 2 (execute step 1): fetch campaign templates → shape proposal
  turn 3 (execute step 2): fetch voucher pool state → shape proposal
  turn 4 (execute step 3): fetch experiment framework → shape proposal
```

The plan phase would use Sonnet 4.6; the execute phase could use Haiku (cheaper, faster). Estimated savings: ~40% cost on recommendation, roughly ~$0.05 → ~$0.03 per case. Would take a refactor of `lib/agents/recommendation.ts` — the aptkit `RecommendationAgent` class is a ReAct loop; you'd need either an aptkit escape hatch to run planner + executor separately, or a wrapper that calls aptkit once for the plan then N times per step.

**The tradeoff that makes this fragile.** Plans are brittle when execution reveals the plan was wrong. If the plan assumes voucher-pool inventory exists and the execute step finds an empty pool, the whole plan needs revising. Mitigation: a **re-plan trigger** — every execute step reports whether it diverged, and after some divergence threshold, throw the plan away and rebuild. Adds complexity — you now have three modes (planning, executing, re-planning) instead of one.

### Move 3 — the principle

Plan-and-execute buys you predictability at the cost of adaptability. Use it when the task is structured enough that a good plan is worth building. Reach for ReAct when the task is exploratory. Most production systems are ReAct with a hardened termination path; a smaller set of well-defined tasks benefit from the phase split.

## Elaborate

The name came out of the "Plan-and-Solve Prompting" paper (Wang et al., 2023) and was operationalized as an agent architecture by LangChain's PlanAndExecute agent around the same time. The key insight is separating "the hard cognitive work" (planning) from "the mechanical work" (executing). It's a variant of the classical AI planning literature (STRIPS, PDDL) grafted onto LLMs.

The interesting escalation from plan-and-execute is **hierarchical planning** — the top-level plan has coarse steps that are themselves plans generated by sub-agents. That's a bridge into multi-agent territory (SECTION C, supervisor-worker) — you can read supervisor-worker as "hierarchical plan-and-execute where each executor is another agent."

## Interview defense

**Q: Why didn't you use plan-and-execute for the diagnostic agent?**

The diagnostic path is genuinely dynamic — the next EQL to run depends on what the last EQL returned. A plan-and-execute here would build a plan on turn 1 with no data in hand, and the plan would be wrong by turn 2. ReAct's per-turn re-decision fits the exploratory shape.

Where I'd reach for plan-and-execute: the recommendation agent. Once the diagnosis is settled, the job is more structured — "propose N Bloomreach actions that match cause X." That could use Sonnet to plan the three action types up front, then Haiku to shape each. I estimate ~40% cost savings; haven't shipped it because the current cost baseline is already inside budget.

*Anchor visual:* the two-phase diagram above.

**Q: What's the failure mode?**

Stale plans. The model commits to a plan based on assumptions, and execution reveals the assumptions were wrong. The classic mitigation is a re-plan trigger — every execute step reports divergence, and past a threshold you throw the plan away. But now you have three modes (planning, executing, re-planning) and the debugging surface grows.
