# AI features in this codebase

Every LLM-powered feature in blooming insights, with the patterns each uses and why. The codebase is **LLM application engineering** ‚Äî all AI is Claude (via `@anthropic-ai/sdk`) orchestrating read-only tools through a `DataSource` seam (Bloomreach MCP in prod; a Blooming-owned in-process `SyntheticDataSource` for local/test). There are **no machine-learning features** (no trained models, recommenders, or on-device inference), so `ml-features-in-this-codebase.md` is intentionally absent.

Common spine: all five active agents (monitoring, diagnostic, recommendation, query, intent) come from `@aptkit/core@0.3.0` and are wired into this codebase by three Blooming-owned adapter classes in `lib/agents/aptkit-adapters.ts` (`AnthropicModelProviderAdapter`, `BloomingToolRegistryAdapter`, `BloomingTraceSinkAdapter`). Models: agents run `claude-sonnet-4-6` (`AGENT_MODEL`, `base.ts:10`); the intent classifier runs `claude-haiku-4-5-20251001` (`intent.ts:14`). No sampling parameters are tuned (Claude defaults); only `max_tokens` is set per call. Tests: **221** Vitest unit tests. The 4-pillar eval suite under `eval/` (detection / diagnosis / recommendation / regression) that used to layer on top was removed in PR #8 (commit 62c24d7) ‚Äî there is no live-agent quality harness in the repo right now. Legacy fully-Blooming-authored implementations live alongside the active ones under `lib/agents/*-legacy.ts` and `lib/agents/legacy-prompts/` ‚Äî preserved-not-active, honest naming.

### Monitoring agent ‚Äî the morning briefing

- **Feature:** Scans the workspace for significant recent changes and emits the most significant anomalies as ranked insight cards (`MonitoringAgent.scan`, `lib/agents/monitoring.ts:69‚Äì120`). `scan(hooks?, categories: AnomalyCategory[] = [])` now takes a category checklist and builds it into the prompt (`{categories}` slot, `monitoring.ts:73‚Äì86`); the briefing route gates that list to the schema-runnable categories (see the schema-gate entry below).
- **Patterns used:** `04-agents-and-tool-use/07-capability-gating.md` (the runnable-category checklist is computed by the gate before the agent runs), `04-agents-and-tool-use/02-tool-calling.md` (scoped `monitoringTools`, `maxToolCalls: 6`), `01-llm-foundations/04-structured-outputs.md` (`isAnomalyArray` validator), `01-llm-foundations/02-tokenization.md` + `02-context-and-prompts/01-context-window.md` (`schemaSummary` char caps: `MAX_EVENTS = 20` / `MAX_PROPS_PER_EVENT = 10` / `MAX_CPROPS = 30` at `monitoring.ts:21/22/33`), `04-agents-and-tool-use/06-error-recovery.md` (returns `[]` on parse failure; `SEV_RANK` sort + top-10 slice at `monitoring.ts:51/119`).
- **Why these patterns:** the workspace schema is large and the rate limit is tight, so the agent gets a truncated schema + a small tool budget + a schema-gated checklist, and must return a validated array or nothing ‚Äî no fabricated alerts, no queries on absent data.

### Anomaly-coverage schema gate ‚Äî scope before spend

- **Feature:** Before the monitoring agent runs, a cheap in-memory gate classifies a fixed 10-category anomaly checklist against the live workspace schema and hands the agent only the runnable categories (`lib/agents/categories.ts`: `schemaCapabilities` `:116‚Äì127` ‚Üí `coverageFor` `:131‚Äì136` ‚Üí `coverageReport` `:144‚Äì155` / `runnableCategories` `:158‚Äì160`; wired in `app/api/briefing/route.ts:202‚Äì204`, fed to `agent.scan(hooks, runnable)` at `:223`). The full report is streamed per-category to the feed's coverage grid (`coverage_item` events at `briefing/route.ts:209‚Äì212`); unsupported categories render as honest "no data source" ghost tiles.
- **Patterns used:** `04-agents-and-tool-use/07-capability-gating.md` (the full treatment), `06-production-serving/02-llm-cost-optimization.md` (gate-before-spend is a budget lever ‚Äî the agent never spends a rate-limited call on a category the schema can't support), `04-agents-and-tool-use/04-tool-routing.md` (a third narrowing of the agent's surface, by data availability rather than agent role).
- **Why these patterns:** the agent's scarce resource is its ~1 req/s, `maxToolCalls`-capped budget; a free schema check up front means every spent call is on a category that can produce a result, and the same `coverageReport` doubles as the UI's coverage state.

### Diagnostic agent ‚Äî why it happened

- **Feature:** Investigates one anomaly and produces a grounded `Diagnosis` (conclusion + evidence + hypotheses considered + optional `affectedCustomers`/`confidence`/`timeSeries`), streaming its reasoning live (`DiagnosticAgent.investigate`, `lib/agents/diagnostic.ts:45‚Äì83`).
- **Patterns used:** `04-agents-and-tool-use/03-react-pattern.md` (thought‚Üítool‚Üíobservation, streamed), `01-llm-foundations/04-structured-outputs.md` (`isDiagnosis` + a dedicated tool-less `synthesize()` at `diagnostic.ts:87‚Äì126`), `05-evals-and-observability/04-llm-observability.md` (a `confidence` is derived from how thoroughly hypotheses were tested via `diagnosisConfidence`, then downgraded if any tool call errored ‚Äî `diagnostic.ts:80‚Äì82`), `04-agents-and-tool-use/06-error-recovery.md` (`FALLBACK` diagnosis at `diagnostic.ts:16‚Äì19`).
- **Why these patterns:** the model wants to keep querying; the forced synthesis turn (and the separate `synthesize()` retry on a clean context) guarantees a conclusion grounded in the evidence it already gathered, and the derived `confidence` keeps the surfaced trust signal honest about the data actually obtained.

### Recommendation agent ‚Äî what to do

- **Feature:** Turns an anomaly + diagnosis into up to three concrete Bloomreach actions, each tagged with the feature it uses and an `estimatedImpact` that may be a plain string or a richer `{ range, rangeUsd?, assumption }` object (`RecommendationAgent.propose`, `lib/agents/recommendation.ts:36‚Äì77`).
- **Patterns used:** `04-agents-and-tool-use/02-tool-calling.md` (scoped `recommendationTools`), `01-llm-foundations/04-structured-outputs.md` (`isRecommendationArray` validates the id-less shape including the `EstimatedImpact` union and optional `effort`/`timeToSetUpMinutes`/`prerequisites`/`successMetric` fields; code assigns `crypto.randomUUID()` at `recommendation.ts:76`, caps at 3), `02-context-and-prompts/03-prompt-chaining.md` (runs after the diagnosis ‚Äî the diagnosis is the recommendation step's input), dedicated `synthesize()` (`recommendation.ts:82‚Äì132`).
- **Why these patterns:** recommendations must be actionable and bounded; validating an id-less array then assigning ids server-side keeps the model's job small and the output trustworthy, while the optional rich fields let the UI show effort/setup-time without forcing the model to always produce them.

### Query agent ‚Äî ask anything

- **Feature:** Answers a free-form natural-language question about the workspace with a tool-grounded answer (`QueryAgent.answer`, `lib/agents/query.ts:24‚Äì47`).
- **Patterns used:** `04-agents-and-tool-use/04-tool-routing.md` (the full tool set + intent routing in front), `01-llm-foundations/05-streaming.md` (NDJSON answer), `04-agents-and-tool-use/05-agent-memory.md` (one-shot ‚Äî `userPrompt: query` at `query.ts:35`, no conversation memory yet), `06-production-serving/03-prompt-injection.md` (the `?q=` input is only `.trim()`'d ‚Äî an open injection surface, bounded by read-only tools).
- **Why these patterns:** reuse the same loop with the broadest tool set; the open input is the one place an adversary can reach the model, so it is flagged as the priority hardening target.

### Intent classifier ‚Äî routing the question

- **Feature:** Decides which agent path a `?q=` question takes (`parseIntent` heuristic at `intent.ts:6‚Äì12`, then `classifyIntent` haiku at `intent.ts:17‚Äì31`).
- **Patterns used:** `01-llm-foundations/07-heuristic-before-llm.md` (cheap deterministic check first), `01-llm-foundations/06-token-economics.md` (haiku + `max_tokens: 16` forces a one-word answer), `04-agents-and-tool-use/04-tool-routing.md`.
- **Why these patterns:** classification is high-volume and cheap-to-get-right; a heuristic resolves the obvious cases for free and only ambiguous ones pay for the (already cheap) model.

### Structured-output extraction + validation

- **Feature:** Converts every agent's prose into a typed contract or rejects it (`parseAgentJson` + type guards, `lib/mcp/validate.ts:3‚Äì53`).
- **Patterns used:** `01-llm-foundations/04-structured-outputs.md`, `01-llm-foundations/01-what-an-llm-is.md` (treat output as untrusted), `04-agents-and-tool-use/06-error-recovery.md`.
- **Why these patterns:** the model is a next-token function; a parse-then-validate boundary is the same discipline as validating an API response before trusting it.

### Streaming reasoning trace (observability-as-product)

- **Feature:** Streams `reasoning_step` / `tool_call_start` / `tool_call_end` / `diagnosis` / `recommendation` events so the user watches the agents work, and reuses the same wire format to replay cached investigations (`AgentEvent` in `lib/mcp/events.ts`, emitted from `app/api/agent/route.ts`, consumed by `lib/hooks/useInvestigation.ts`). The trace UI renders each line with a `toLocaleTimeString` clock and tool durations (`components/shared/StatusLog.tsx` ‚Üí `components/investigation/ReasoningTrace.tsx`, `ts` stamped at `useInvestigation.ts:106/113`) and pretty-prints fenced JSON / markdown content (`components/investigation/TraceContent.tsx`).
- **Patterns used:** `01-llm-foundations/05-streaming.md`, `05-evals-and-observability/04-llm-observability.md`, `04-agents-and-tool-use/03-react-pattern.md`.
- **Why these patterns:** "show its work" is both the UX and the telemetry; the same events that build the UI are the timestamped trace an engineer would inspect, and the wire format doubles as the replay format for cached snapshots.

### Two-step investigation (diagnose ‚Üí recommend) with a sessionStorage handoff

- **Feature:** The drill-down is split into two pages and two separate stream calls: step 2 (`app/investigate/[id]/page.tsx`) runs ONLY the diagnostic agent (`/api/agent?step=diagnose`); step 3 (`app/investigate/[id]/recommend/page.tsx`) runs ONLY the recommendation agent (`/api/agent?step=recommend`), fed the diagnosis handed over from step 2. The split is driven by `useInvestigation(id, step)` (`lib/hooks/useInvestigation.ts`); the route reads the `step` param at `route.ts:117‚Äì118`, picks the lead agent at `route.ts:199‚Äì200`, and gates each agent (`route.ts:225‚Äì249`). The diagnose step writes the diagnosis to `sessionStorage` under `bi:diag:<id>` (`useInvestigation.ts:138‚Äì139`); the recommend step reads it back and forwards it as `&diagnosis=` (`useInvestigation.ts:72‚Äì77, 162‚Äì163`). A null `step` is the legacy combined run.
- **Patterns used:** `04-agents-and-tool-use/01-agents-vs-chains.md` (each step is one agent, not the whole pipeline), `02-context-and-prompts/03-prompt-chaining.md` (the diagnosis is the literal input to the recommendation step), `01-llm-foundations/05-streaming.md` (each step is its own NDJSON stream consumed by the hook).
- **Why these patterns:** splitting the chain across two user-driven steps means the expensive recommendation run is deferred until the user actually asks for it, and the sessionStorage handoff carries the prior step's output across the page navigation without re-running the diagnostic agent.

### Demo vs. live mode + the dev one-click demo-snapshot capture

- **Feature:** The feed runs in two runtime modes toggled in the UI and persisted in `localStorage` (`app/page.tsx:108‚Äì131`): **demo** replays a committed snapshot (`/api/briefing?demo=cached` ‚Üí `lib/state/demo-insights.json`, served at `briefing/route.ts:42‚Äì50`), **live** runs the agents against the real workspace. A dev-only "capture this as the demo snapshot (one click)" button (`app/page.tsx:196‚Äì246`, gated to `NODE_ENV !== 'production'` and live mode) POSTs the current live briefing to `/api/mcp/capture-demo` (writes `demo-insights.json` + bundles already-run investigations into `demo-investigations.json`); the route itself is hard-disabled in production (`capture-demo/route.ts:11‚Äì14`).
- **Patterns used:** `05-evals-and-observability/04-llm-observability.md` (a captured snapshot is a frozen, replayable trace ‚Äî a de-facto golden record of one real run), `06-production-serving/01-llm-caching.md` (demo mode is a whole-response cache that needs no API key and is instant).
- **Why these patterns:** a live multi-agent run costs ~100s and an API key; baking one real run into a committed snapshot makes the demo instant and reliable for a presentation while keeping the capture path dev-only because serverless filesystems are read-only.

### Enrichment derivation (agent-written impact + derived confidence)

- **Feature:** Two enrichments make agent output business-readable without asking the model to fabricate. (1) The monitoring agent writes a one-sentence business `impact` per anomaly, carried straight through `anomalyToInsight` (`lib/state/insights.ts:23`); alongside it, `deriveInsightFields` (`lib/insights/derive.ts:27‚Äì39`) computes a `revenueImpact` purely from the evidence the agent already gathered (no new data). (2) The diagnostic agent's `confidence` is **derived** by `diagnosisConfidence` (`lib/insights/derive.ts:54‚Äì63`) from how thoroughly hypotheses were tested and whether one was supported ‚Äî preferring the agent's own `confidence` when set, and downgraded high‚Üímedium when any tool call errored (`diagnostic.ts:80‚Äì82`).
- **Patterns used:** `01-llm-foundations/04-structured-outputs.md` (the agent emits a typed `impact`; the derived fields are optional + computed, so older snapshots still validate), `05-evals-and-observability/04-llm-observability.md` (a confidence derived from observed hypothesis-testing is an honest trust signal, not a self-reported one).
- **Why these patterns:** asking the model for a self-reported confidence invites optimism; deriving it from what was actually tested ‚Äî and downgrading it when queries failed ‚Äî keeps the surfaced signal grounded in the run's real evidence, while keeping the derived fields optional preserves backward compatibility.

### DataSource adapter seam ‚Äî backend-swappable agents

- **Feature:** Every agent depends on a `DataSource` interface (`lib/data-source/types.ts` ‚Äî `callTool` / `listTools` / `dispose`) instead of the concrete `McpClient`. Two adapters implement it: `BloomreachDataSource` (`bloomreach-data-source.ts`, wraps the existing `McpClient` over the live Bloomreach OAuth + MCP transport) and `SyntheticDataSource` (`synthetic-data-source.ts`, a Blooming-owned in-process fake returning canned ecommerce events for the synthetic workspace ‚Äî no subprocess, no SQL). The route picks one via `makeDataSource(mode, sessionId)` (`lib/data-source/index.ts`) reading `bi:mode` = `'demo'` | `'live-synthetic'` | `'live-bloomreach'` from the client.
- **Patterns used:** `01-llm-foundations/08-provider-abstraction.md` (this is the *data*-provider abstraction; the *model*-provider is still single-vendor Anthropic), `04-agents-and-tool-use/02-tool-calling.md` (the `McpCaller` boundary inside the agent loop is satisfied by any `DataSource`), `04-agents-and-tool-use/09-aptkit-primitive-adapters.md` (the `BloomingToolRegistryAdapter` is what bridges any `DataSource` into AptKit's `ToolRegistry` primitive).
- **Why these patterns:** keeping the agent surface dependent on a `DataSource` interface means the route never knows which protocol is on the other side. The synthetic adapter is a deterministic substrate for local development without OAuth + Bloomreach rate limits; the Bloomreach adapter is what production uses.

### AptKit primitive adapters ‚Äî own your domain glue, use a library's generic primitives

- **Feature:** Five active agents (`AnomalyMonitoringAgent`, `DiagnosticAgent`, `RecommendationAgent`, `QueryAgent`, intent classifier) come from `@aptkit/core@0.3.0`. Three Blooming-owned adapter classes in `lib/agents/aptkit-adapters.ts` (206 LOC) bridge Blooming runtime objects to AptKit's provider-neutral primitives: `AnthropicModelProviderAdapter` wraps `@anthropic-ai/sdk` as AptKit's `ModelProvider`; `BloomingToolRegistryAdapter` wraps a `DataSource` as AptKit's `ToolRegistry`; `BloomingTraceSinkAdapter` consumes AptKit's `CapabilityTraceSink` events and re-emits them through Blooming's existing NDJSON `onText` / `onToolCall` / `onToolResult` hooks. Active prompts ship via `@aptkit/prompts`; active output validators ship via `@aptkit/agent-*` validators. The fully-Blooming-authored predecessor is preserved-not-active under `lib/agents/*-legacy.ts`, `lib/agents/legacy-prompts/`, and `lib/agents/legacy-validate.ts`.
- **Patterns used:** `04-agents-and-tool-use/09-aptkit-primitive-adapters.md` (the full treatment), `01-llm-foundations/08-provider-abstraction.md` (each adapter is a provider-abstraction seam ‚Äî model provider, tool registry, trace sink), `04-agents-and-tool-use/02-tool-calling.md` (the loop now lives in `@aptkit/core`, called through the registry adapter).
- **Why these patterns:** the senior-level pattern is "use the library's generic primitives + own the adapters into your domain." The loop logic, the tool-use protocol, and the validators are not unique to Blooming ‚Äî they belong in a shared library (`@aptkit/core`). The Anthropic-vs-generic boundary, the DataSource-vs-ToolRegistry boundary, and the NDJSON-vs-CapabilityEvent boundary all belong to Blooming ‚Äî they encode the specific infra this app already has. Mixing the two layers (Blooming-owned loop, or library-owned NDJSON shape) is what makes integrations brittle. The legacy/active split is the honest record of the migration.
