# AI features in this codebase

Every LLM-powered feature in blooming insights, with the patterns each uses and why. The codebase is **LLM application engineering** — all AI is Claude (via `@anthropic-ai/sdk`) orchestrating read-only tools through a `DataSource` seam (Bloomreach MCP in prod; an authored SQLite-backed `mcp-server-olist/` subprocess in eval/local). There are **no machine-learning features** (no trained models, recommenders, or on-device inference), so `ml-features-in-this-codebase.md` is intentionally absent.

Common spine: all four agents share `runAgentLoop` (`lib/agents/base.ts`) — a Claude tool-use loop bounded by a `maxToolCalls` budget that forces a tool-less final turn carrying a `synthesisInstruction`. Models: agents run `claude-sonnet-4-6` (`AGENT_MODEL`, `base.ts:10`); the intent classifier runs `claude-haiku-4-5-20251001` (`intent.ts:14`). No sampling parameters are tuned (Claude defaults); only `max_tokens` is set per call. Tests: **269** Vitest unit tests (up from 144), plus the four end-to-end evals under `eval/scripts/` that exercise the real Sonnet 4.6 agents over the live OlistDataSource.

### Monitoring agent — the morning briefing

- **Feature:** Scans the workspace for significant recent changes and emits the most significant anomalies as ranked insight cards (`MonitoringAgent.scan`, `lib/agents/monitoring.ts:69–120`). `scan(hooks?, categories: AnomalyCategory[] = [])` now takes a category checklist and builds it into the prompt (`{categories}` slot, `monitoring.ts:73–86`); the briefing route gates that list to the schema-runnable categories (see the schema-gate entry below).
- **Patterns used:** `04-agents-and-tool-use/07-capability-gating.md` (the runnable-category checklist is computed by the gate before the agent runs), `04-agents-and-tool-use/02-tool-calling.md` (scoped `monitoringTools`, `maxToolCalls: 6`), `01-llm-foundations/04-structured-outputs.md` (`isAnomalyArray` validator), `01-llm-foundations/02-tokenization.md` + `02-context-and-prompts/01-context-window.md` (`schemaSummary` char caps: `MAX_EVENTS = 20` / `MAX_PROPS_PER_EVENT = 10` / `MAX_CPROPS = 30` at `monitoring.ts:21/22/33`), `04-agents-and-tool-use/06-error-recovery.md` (returns `[]` on parse failure; `SEV_RANK` sort + top-10 slice at `monitoring.ts:51/119`).
- **Why these patterns:** the workspace schema is large and the rate limit is tight, so the agent gets a truncated schema + a small tool budget + a schema-gated checklist, and must return a validated array or nothing — no fabricated alerts, no queries on absent data.

### Anomaly-coverage schema gate — scope before spend

- **Feature:** Before the monitoring agent runs, a cheap in-memory gate classifies a fixed 10-category anomaly checklist against the live workspace schema and hands the agent only the runnable categories (`lib/agents/categories.ts`: `schemaCapabilities` `:116–127` → `coverageFor` `:131–136` → `coverageReport` `:144–155` / `runnableCategories` `:158–160`; wired in `app/api/briefing/route.ts:202–204`, fed to `agent.scan(hooks, runnable)` at `:223`). The full report is streamed per-category to the feed's coverage grid (`coverage_item` events at `briefing/route.ts:209–212`); unsupported categories render as honest "no data source" ghost tiles.
- **Patterns used:** `04-agents-and-tool-use/07-capability-gating.md` (the full treatment), `06-production-serving/02-llm-cost-optimization.md` (gate-before-spend is a budget lever — the agent never spends a rate-limited call on a category the schema can't support), `04-agents-and-tool-use/04-tool-routing.md` (a third narrowing of the agent's surface, by data availability rather than agent role).
- **Why these patterns:** the agent's scarce resource is its ~1 req/s, `maxToolCalls`-capped budget; a free schema check up front means every spent call is on a category that can produce a result, and the same `coverageReport` doubles as the UI's coverage state.

### Diagnostic agent — why it happened

- **Feature:** Investigates one anomaly and produces a grounded `Diagnosis` (conclusion + evidence + hypotheses considered + optional `affectedCustomers`/`confidence`/`timeSeries`), streaming its reasoning live (`DiagnosticAgent.investigate`, `lib/agents/diagnostic.ts:45–83`).
- **Patterns used:** `04-agents-and-tool-use/03-react-pattern.md` (thought→tool→observation, streamed), `01-llm-foundations/04-structured-outputs.md` (`isDiagnosis` + a dedicated tool-less `synthesize()` at `diagnostic.ts:87–126`), `05-evals-and-observability/04-llm-observability.md` (a `confidence` is derived from how thoroughly hypotheses were tested via `diagnosisConfidence`, then downgraded if any tool call errored — `diagnostic.ts:80–82`), `04-agents-and-tool-use/06-error-recovery.md` (`FALLBACK` diagnosis at `diagnostic.ts:16–19`).
- **Why these patterns:** the model wants to keep querying; the forced synthesis turn (and the separate `synthesize()` retry on a clean context) guarantees a conclusion grounded in the evidence it already gathered, and the derived `confidence` keeps the surfaced trust signal honest about the data actually obtained.

### Recommendation agent — what to do

- **Feature:** Turns an anomaly + diagnosis into up to three concrete Bloomreach actions, each tagged with the feature it uses and an `estimatedImpact` that may be a plain string or a richer `{ range, rangeUsd?, assumption }` object (`RecommendationAgent.propose`, `lib/agents/recommendation.ts:36–77`).
- **Patterns used:** `04-agents-and-tool-use/02-tool-calling.md` (scoped `recommendationTools`), `01-llm-foundations/04-structured-outputs.md` (`isRecommendationArray` validates the id-less shape including the `EstimatedImpact` union and optional `effort`/`timeToSetUpMinutes`/`prerequisites`/`successMetric` fields; code assigns `crypto.randomUUID()` at `recommendation.ts:76`, caps at 3), `02-context-and-prompts/03-prompt-chaining.md` (runs after the diagnosis — the diagnosis is the recommendation step's input), dedicated `synthesize()` (`recommendation.ts:82–132`).
- **Why these patterns:** recommendations must be actionable and bounded; validating an id-less array then assigning ids server-side keeps the model's job small and the output trustworthy, while the optional rich fields let the UI show effort/setup-time without forcing the model to always produce them.

### Query agent — ask anything

- **Feature:** Answers a free-form natural-language question about the workspace with a tool-grounded answer (`QueryAgent.answer`, `lib/agents/query.ts:24–47`).
- **Patterns used:** `04-agents-and-tool-use/04-tool-routing.md` (the full tool set + intent routing in front), `01-llm-foundations/05-streaming.md` (NDJSON answer), `04-agents-and-tool-use/05-agent-memory.md` (one-shot — `userPrompt: query` at `query.ts:35`, no conversation memory yet), `06-production-serving/03-prompt-injection.md` (the `?q=` input is only `.trim()`'d — an open injection surface, bounded by read-only tools).
- **Why these patterns:** reuse the same loop with the broadest tool set; the open input is the one place an adversary can reach the model, so it is flagged as the priority hardening target.

### Intent classifier — routing the question

- **Feature:** Decides which agent path a `?q=` question takes (`parseIntent` heuristic at `intent.ts:6–12`, then `classifyIntent` haiku at `intent.ts:17–31`).
- **Patterns used:** `01-llm-foundations/07-heuristic-before-llm.md` (cheap deterministic check first), `01-llm-foundations/06-token-economics.md` (haiku + `max_tokens: 16` forces a one-word answer), `04-agents-and-tool-use/04-tool-routing.md`.
- **Why these patterns:** classification is high-volume and cheap-to-get-right; a heuristic resolves the obvious cases for free and only ambiguous ones pay for the (already cheap) model.

### Structured-output extraction + validation

- **Feature:** Converts every agent's prose into a typed contract or rejects it (`parseAgentJson` + type guards, `lib/mcp/validate.ts:3–53`).
- **Patterns used:** `01-llm-foundations/04-structured-outputs.md`, `01-llm-foundations/01-what-an-llm-is.md` (treat output as untrusted), `04-agents-and-tool-use/06-error-recovery.md`.
- **Why these patterns:** the model is a next-token function; a parse-then-validate boundary is the same discipline as validating an API response before trusting it.

### Streaming reasoning trace (observability-as-product)

- **Feature:** Streams `reasoning_step` / `tool_call_start` / `tool_call_end` / `diagnosis` / `recommendation` events so the user watches the agents work, and reuses the same wire format to replay cached investigations (`AgentEvent` in `lib/mcp/events.ts`, emitted from `app/api/agent/route.ts`, consumed by `lib/hooks/useInvestigation.ts`). The trace UI renders each line with a `toLocaleTimeString` clock and tool durations (`components/shared/StatusLog.tsx` → `components/investigation/ReasoningTrace.tsx`, `ts` stamped at `useInvestigation.ts:106/113`) and pretty-prints fenced JSON / markdown content (`components/investigation/TraceContent.tsx`).
- **Patterns used:** `01-llm-foundations/05-streaming.md`, `05-evals-and-observability/04-llm-observability.md`, `04-agents-and-tool-use/03-react-pattern.md`.
- **Why these patterns:** "show its work" is both the UX and the telemetry; the same events that build the UI are the timestamped trace an engineer would inspect, and the wire format doubles as the replay format for cached snapshots.

### Two-step investigation (diagnose → recommend) with a sessionStorage handoff

- **Feature:** The drill-down is split into two pages and two separate stream calls: step 2 (`app/investigate/[id]/page.tsx`) runs ONLY the diagnostic agent (`/api/agent?step=diagnose`); step 3 (`app/investigate/[id]/recommend/page.tsx`) runs ONLY the recommendation agent (`/api/agent?step=recommend`), fed the diagnosis handed over from step 2. The split is driven by `useInvestigation(id, step)` (`lib/hooks/useInvestigation.ts`); the route reads the `step` param at `route.ts:117–118`, picks the lead agent at `route.ts:199–200`, and gates each agent (`route.ts:225–249`). The diagnose step writes the diagnosis to `sessionStorage` under `bi:diag:<id>` (`useInvestigation.ts:138–139`); the recommend step reads it back and forwards it as `&diagnosis=` (`useInvestigation.ts:72–77, 162–163`). A null `step` is the legacy combined run.
- **Patterns used:** `04-agents-and-tool-use/01-agents-vs-chains.md` (each step is one agent, not the whole pipeline), `02-context-and-prompts/03-prompt-chaining.md` (the diagnosis is the literal input to the recommendation step), `01-llm-foundations/05-streaming.md` (each step is its own NDJSON stream consumed by the hook).
- **Why these patterns:** splitting the chain across two user-driven steps means the expensive recommendation run is deferred until the user actually asks for it, and the sessionStorage handoff carries the prior step's output across the page navigation without re-running the diagnostic agent.

### Demo vs. live mode + the dev one-click demo-snapshot capture

- **Feature:** The feed runs in two runtime modes toggled in the UI and persisted in `localStorage` (`app/page.tsx:108–131`): **demo** replays a committed snapshot (`/api/briefing?demo=cached` → `lib/state/demo-insights.json`, served at `briefing/route.ts:42–50`), **live** runs the agents against the real workspace. A dev-only "capture this as the demo snapshot (one click)" button (`app/page.tsx:196–246`, gated to `NODE_ENV !== 'production'` and live mode) POSTs the current live briefing to `/api/mcp/capture-demo` (writes `demo-insights.json` + bundles already-run investigations into `demo-investigations.json`); the route itself is hard-disabled in production (`capture-demo/route.ts:11–14`).
- **Patterns used:** `05-evals-and-observability/04-llm-observability.md` (a captured snapshot is a frozen, replayable trace — a de-facto golden record of one real run), `06-production-serving/01-llm-caching.md` (demo mode is a whole-response cache that needs no API key and is instant).
- **Why these patterns:** a live multi-agent run costs ~100s and an API key; baking one real run into a committed snapshot makes the demo instant and reliable for a presentation while keeping the capture path dev-only because serverless filesystems are read-only.

### Enrichment derivation (agent-written impact + derived confidence)

- **Feature:** Two enrichments make agent output business-readable without asking the model to fabricate. (1) The monitoring agent writes a one-sentence business `impact` per anomaly, carried straight through `anomalyToInsight` (`lib/state/insights.ts:23`); alongside it, `deriveInsightFields` (`lib/insights/derive.ts:27–39`) computes a `revenueImpact` purely from the evidence the agent already gathered (no new data). (2) The diagnostic agent's `confidence` is **derived** by `diagnosisConfidence` (`lib/insights/derive.ts:54–63`) from how thoroughly hypotheses were tested and whether one was supported — preferring the agent's own `confidence` when set, and downgraded high→medium when any tool call errored (`diagnostic.ts:80–82`).
- **Patterns used:** `01-llm-foundations/04-structured-outputs.md` (the agent emits a typed `impact`; the derived fields are optional + computed, so older snapshots still validate), `05-evals-and-observability/04-llm-observability.md` (a confidence derived from observed hypothesis-testing is an honest trust signal, not a self-reported one).
- **Why these patterns:** asking the model for a self-reported confidence invites optimism; deriving it from what was actually tested — and downgrading it when queries failed — keeps the surfaced signal grounded in the run's real evidence, while keeping the derived fields optional preserves backward compatibility.

### DataSource adapter seam — backend-swappable agents

- **Feature:** Every agent depends on a `DataSource` interface (`lib/data-source/types.ts` — `callTool` / `listTools` / `dispose`) instead of the concrete `McpClient`. Two adapters implement it: `BloomreachDataSource` (`bloomreach-data-source.ts`, wraps the existing `McpClient` over the live Bloomreach OAuth + MCP transport) and `OlistDataSource` (`olist-data-source.ts`, spawns the sibling `mcp-server-olist/` Node subprocess over stdio). The route picks one via `makeDataSource(mode, sessionId)` (`lib/data-source/index.ts`) reading `bi:mode` = `'demo'` | `'live-sql'` | `'live-bloomreach'` from the client.
- **Patterns used:** `01-llm-foundations/08-provider-abstraction.md` (this is the *data*-provider abstraction; the *model*-provider is still single-vendor Anthropic), `04-agents-and-tool-use/02-tool-calling.md` (the `McpCaller` boundary inside `runAgentLoop` is satisfied by any `DataSource`), `04-agents-and-tool-use/08-authoring-mcp-server.md` (the Olist adapter is what made authoring a sibling MCP server worthwhile in the first place).
- **Why these patterns:** the eval pillar needs deterministic, seedable data with ground-truth anomalies — Bloomreach can't give you either — so the agents had to grow a seam. The same seam keeps `live-bloomreach` working for real demos, and the route never knows which protocol is on the other side.

### Authored Olist MCP server — domain tools, not raw EQL

- **Feature:** A sibling Node package, `mcp-server-olist/` (~1800 LOC), exposes **three domain tools** over a synthetic Brazilian e-commerce SQLite dataset (~10k orders, 26-week horizon, **three seeded ground-truth anomalies** in a `seeded_anomalies` table — `sp-revenue-drop-w4`, `electronics-spike-w2`, `voucher-dropoff-w10-on`): `get_metric_timeseries` (`src/tools/get_metric_timeseries.ts`), `get_segments` (`src/tools/get_segments.ts`), `get_anomaly_context` (`src/tools/get_anomaly_context.ts`). It is **not** a raw `execute_sql` surface — the agent gets compiled, parameterized analytics primitives instead of an open SQL prompt.
- **Patterns used:** `04-agents-and-tool-use/08-authoring-mcp-server.md` (the full treatment), `04-agents-and-tool-use/02-tool-calling.md` (these tools join the per-agent `monitoringTools` / `diagnosticTools` / `recommendationTools` sets at `lib/mcp/tools.ts:49–53`), `06-production-serving/03-prompt-injection.md` (domain tools are a much smaller attack surface than `execute_sql` — there is no model-authored SQL to escape).
- **Why these patterns:** raw SQL tools let the model write arbitrary queries, including bad ones (typos, wrong joins, expensive scans, accidental injection vectors). Domain tools shift that burden into authored TypeScript — every query the agent can run is one you wrote, tested, and can reason about. The cost is more authoring up front; the win is a much smaller, much safer surface that the eval suite can score deterministically.

### Prompt anchoring — DATA HORIZON + 3-dimensional scan plan

- **Feature:** Every long-running prompt — `lib/agents/prompts/monitoring.md`, `diagnostic.md`, `query.md`, plus the recommendation prompt — carries a **DATA HORIZON section** that anchors `time_range` arguments to the synthetic dataset's actual window (`2025-12-01 → 2026-06-01`), and the monitoring prompt adds a **3-dimensional scan plan** (state × category × payment_type) for the Olist core metrics. The horizon is also injected into the workspace schema at the bottom of each prompt via `schemaSummary()` (`lib/mcp/schema.ts` emits `Data horizon: <from> → <to>`).
- **Patterns used:** `02-context-and-prompts/01-context-window.md` (the anchor is a few hundred extra tokens that fence off entire failure modes — the agent stops issuing 2017/2018 queries that would return empty), `05-evals-and-observability/02-eval-methods.md` (this whole anchor was reverse-engineered from a detection eval: pre-fix loose recall 6.7% → post-fix 33.3% — a 5× lift), `01-llm-foundations/01-what-an-llm-is.md` (the model has no idea the dataset starts in Dec 2025 unless you tell it — naked schema isn't enough).
- **Why these patterns:** PR D's detection eval surfaced that the agent was issuing time queries outside the dataset horizon, wasting tool calls on guaranteed-empty results. Phase 2.5 added DATA HORIZON; the next eval run lifted loose recall 5× *but* left strict recall at 0% — the "recent 4w vs baseline 12w" framing still cannot catch mid-horizon week-specific anomalies (weeks 2 and 4). This is the **honest "the prompt fix is partial"** finding — documented in `eval/results/2026-06-15-after-fix/summary.md` and exactly the kind of evidence-driven prompt iteration the eval flywheel is supposed to produce.

### Detection eval — does the monitoring agent find the seeded anomalies

- **Feature:** `eval/scripts/run-detection.ts` (+ `eval/scripts/lib/run-agent.ts`, `scorer.ts`, `summary.ts`) spawns one fresh `mcp-server-olist` subprocess per run, invokes the real `MonitoringAgent.scan` against the 3 seeded anomalies K=10 times, and scores each run under two matchers: **LOOSE** (2-of-3 criteria: metric + segment + time-window) and **STRICT** (3-of-3). Aggregates land in `eval/results/<YYYY-MM-DD>/detection-K10-{loose,strict,raw}.json` + a human-readable `summary.md`.
- **Patterns used:** `05-evals-and-observability/01-eval-set-types.md` (golden = the 3 seeded anomalies; this script *is* the golden harness for monitoring), `05-evals-and-observability/02-eval-methods.md` (set-overlap precision/recall — F1 is implicit), `04-agents-and-tool-use/06-error-recovery.md` (each run is in a try/catch and crashes record as `runs[i].error` so the K-series continues honestly).
- **Why these patterns:** monitoring's output is a set, not a label, so set-overlap is the right rung; loose-vs-strict separation is honest (loose is the optimistic ceiling — "did the agent surface something semantically near"; strict is the recruiter number — "exactly this metric, this segment, this time"). One subprocess per run isolates each run's data state from the next.

### Diagnosis eval — is the diagnostic reasoning right

- **Feature:** `eval/scripts/run-diagnosis.ts` (+ `eval/scripts/lib/judge.ts`, `run-diagnostic-agent.ts`) drives the real `DiagnosticAgent` directly on each seeded anomaly's metadata (Path C — bypasses upstream detection so diagnosis quality is isolated), K=10 runs per anomaly, then scores each candidate with a Sonnet-4.6 LLM-as-judge over a **5-criterion rubric** in `eval/judges/diagnosis-judge.md`: hypothesis 0-2, evidence 0-2, sizing 0-2, calibration 0-1, fabrication 0-2 (total 0-9; pass ≥ 7). Results: `eval/results/2026-06-15/diagnosis-K10-{candidates,judge,summary}.json` + `diagnosis-summary.md`. Portfolio number: **53.3% pass-rate, mean 6.37/9**.
- **Patterns used:** `05-evals-and-observability/02-eval-methods.md` (rubric grading is the right rung for structured prose — `Diagnosis` decomposes naturally into hypothesis/evidence/sizing/calibration/fabrication points), `05-evals-and-observability/03-llm-as-judge-bias.md` (verbosity bias controlled by per-criterion scoring; **self-preference bias accepted as a known cost** — judge and agent both run `claude-sonnet-4-6`, calibration receipts are how that cost is held honest).
- **Why these patterns:** the diagnostic agent's output is high-entropy prose with checkable sub-criteria; a per-criterion rubric gives signal a single holistic 1-5 cannot. The Path-C bypass means low detection precision can't dilute the diagnosis number — they are independent quality measurements.

### Recommendation eval — are the proposed actions credible

- **Feature:** `eval/scripts/run-recommendation.ts` invokes `RecommendationAgent.propose` per seeded anomaly with a hand-crafted reference diagnosis as input (same Path-C isolation), K=10 runs, judged by `eval/judges/recommendation-judge.md` under a **3-criterion rubric**: plausible 0-2, specific 0-2, impact_sized 0-1 (total 0-5; pass ≥ 4). Portfolio number: **100% pass-rate, mean 4.97/5** — and the judge is NOT rubber-stamping: at run 8 of electronics-spike-w2 it caught a `R$131,965 AOV → $26K/order` impact figure as a recurrence of the **BRL cents-vs-Reais bug** PR E surfaced, scored `impact_sized=0`, dropping the run to 4/5.
- **Patterns used:** `05-evals-and-observability/02-eval-methods.md` (rubric, not holistic — same reason as diagnosis), `05-evals-and-observability/03-llm-as-judge-bias.md` (calibration receipt: 3/3 manual-vs-judge agreement on a stratified sample including the impact_sized=0 case), `04-agents-and-tool-use/06-error-recovery.md` (the BRL bug is exactly the class of silent regression a regression eval catches recurrence of).
- **Why these patterns:** recommendation output is a small set of structured actions — most of the variance is in the impact numbers, not the action mechanics. The judge catching `R$131,965 AOV` is the proof-of-life for "this judge isn't decorative."

### Regression eval — does today's output match the captured golden

- **Feature:** `eval/scripts/run-regression.ts` runs 10 golden fixtures under `eval/fixtures/regression-golden/` (monitoring × 2, diagnostic × 3, recommendation × 3, query × 1, intent × 1) in two modes: **`capture`** (writes today's outputs into `eval/results/<date>-capture/`) and **`score`** (re-runs the same fixtures against today's code, compares each candidate to its golden via a **two-mode** scorer — structural diff via `eval/scripts/lib/structural-diff.ts` AND a Sonnet-4.6 **similarity judge** at `eval/judges/similarity-judge.md`). Portfolio number: **30% semantic pass-rate baseline** (structural diff = 100% — outputs always have the right shape; the gap is conclusion-level drift).
- **Patterns used:** `05-evals-and-observability/05-regression-evals.md` (the full treatment of the capture/score split + the two-mode scoring), `05-evals-and-observability/01-eval-set-types.md` (this is the regression-set leg of the eval triad), `05-evals-and-observability/03-llm-as-judge-bias.md` (the similarity judge is *also* self-preference-vulnerable — same sonnet-judges-sonnet caveat, same calibration spot-check discipline).
- **Why these patterns:** verbatim equality is the wrong instrument for non-deterministic prose conclusions. Structural diff catches type / required-field regressions for free; the similarity judge catches conclusion-level drift (e.g., "platform-wide surge" flipped to "electronics-specific event" on the same input). The 30% baseline IS the finding — it quantifies how stochastic the agent system actually is.

### Eval flywheel — surfacing real bugs

- **Feature:** The eval pipeline has already caught and documented three real findings the unit tests never would: (1) **PR D's detection eval** surfaced the time-horizon issue → Phase 2.5 prompt anchoring fix → 5× lift in loose recall (6.7% → 33.3%, see `eval/results/2026-06-15-after-fix/summary.md`); (2) **PR E's diagnosis eval** surfaced the BRL cents-vs-Reais unit bug; (3) **PR F's recommendation judge** caught the BRL bug recurring at run 8 of electronics-spike-w2 (R$131,965 AOV → $26K/order), proving the rubric isn't decorative; (4) **PR G's regression baseline** at 30% confirmed conclusion-level instability is a system-wide property, not a one-off.
- **Patterns used:** `05-evals-and-observability/02-eval-methods.md` (the flywheel is "score → surface → fix → re-score"), `05-evals-and-observability/01-eval-set-types.md` (each surfaced bug becomes the next regression fixture).
- **Why these patterns:** an eval pipeline pays for itself only when it catches things unit tests cannot. The paper trail in `eval/results/2026-06-15/`, `…-after-fix/`, `…-capture/`, and `…-score-baseline/` is that proof — dated, committed, and honest about what didn't lift.
