# 05 — Evals and observability

This sub-section is two halves of one question — *how do you know your LLM feature is any good, and how do you see what it did?* — and blooming insights now sits on **the implemented side of both**.

**Observability is a product strength (Case A).** Every investigation already emits a typed trace: `reasoning_step` / `tool_call_start` / `tool_call_end{durationMs}` events (`lib/mcp/events.ts`), streamed live to the UI as the agent's visible work — a sticky `StatusLog` sidebar on both investigate steps, an inline panel on the feed, each line timestamped and the reasoning pretty-printed by `TraceContent`, all fed by the StrictMode-safe `useInvestigation` hook — narrated per call by `describeToolCall` (`app/api/briefing/route.ts`), and replayed event-for-event from the investigation cache (`lib/state/investigations.ts`). The trace is not a backend log — it *is* the product surface. blooming insights built the trace abstraction (spans, durations, replay) by hand.

**Evals are now Case A too.** Phase 3 added a **4-pillar eval suite** under `eval/`: detection (`run-detection.ts` — set-overlap on the 3 seeded anomalies, loose+strict), diagnosis (`run-diagnosis.ts` + `eval/judges/diagnosis-judge.md` — 5-criterion rubric), recommendation (`run-recommendation.ts` + `eval/judges/recommendation-judge.md` — 3-criterion rubric), regression (`run-regression.ts` + `eval/judges/similarity-judge.md` — capture+score two-mode with structural diff and similarity judge). Each runs **K=10** times per anomaly against the real Sonnet-4.6 agents over the authored OlistDataSource; results land in dated dirs (`eval/results/2026-06-15/`, `…-after-fix/`, `…-capture/`, `…-score-baseline/`). The 269 Vitest tests still inject fakes and assert plumbing — they didn't go away. The eval suite is the **separate, parallel, real-money quality measurement** layered on top.

**Portfolio numbers (2026-06-15, K=10):**

| Pillar | Number | What it means |
|---|---|---|
| Detection | 37.0% loose precision, 33.3% loose recall (strict: 0%) | Voucher anomaly perfect; mid-horizon week-specific anomalies still slip past "recent 4w vs baseline 12w" framing |
| Diagnosis | 53.3% pass-rate, mean 6.37/9 | Voucher 10/10; sp-revenue 0/10 — diagnostic reasoning quality is the diagnostic agent's bottleneck |
| Recommendation | 100% pass-rate, mean 4.97/5 | Judge caught BRL cents-vs-Reais bug at run 8 → `impact_sized=0` → rubric not decorative |
| Regression | 30% semantic pass, 100% structural pass | Conclusion-level stochasticity quantified — this is the baseline, not a bug |

- **[01-eval-set-types.md](01-eval-set-types.md)** — *Case A now.* The 269 Vitest tests still inject fakes and assert plumbing, but the **eval sets exist**: golden (the 3 seeded anomalies in `mcp-server-olist/data/olist.db`'s `seeded_anomalies` table + reference diagnoses/recommendations under `eval/fixtures/reference-{diagnoses,recommendations}.json`), regression (`eval/fixtures/regression-golden/` — 10 fixtures captured 2026-06-15). Adversarial coverage on `?q=` is still the named gap. Cross-links the prompt-injection defence in `../06-production-serving/03-prompt-injection.md`.
- **[02-eval-methods.md](02-eval-methods.md)** — *Case A now.* The scoring ladder, **wired**: detection uses set-overlap precision/recall (loose+strict matchers in `eval/scripts/lib/scorer.ts`); diagnosis uses per-criterion rubric scoring via LLM-as-judge (`eval/judges/diagnosis-judge.md` + `eval/scripts/lib/judge.ts`); recommendation uses a smaller per-criterion rubric (`recommendation-judge.md` + `judge-rec.ts`); regression uses two-mode scoring (structural diff via `structural-diff.ts` + similarity judge via `similarity-judge.md`). Pairwise A/B mode is the remaining un-built rung.
- **[03-llm-as-judge-bias.md](03-llm-as-judge-bias.md)** — *Case A — partial; the trap is acknowledged-and-receipted, not avoided.* The judges run `claude-sonnet-4-6` (same as the agents) → **self-preference by construction**. The mitigation that *is* in place: per-criterion rubrics (verbosity-bias control), **manual-vs-judge calibration spot-checks** (diagnosis 8/8 agreement on a stratified sample; recommendation 3/3 including the BRL-bug catch at impact_sized=0). Position bias randomization is N/A for these absolute-score rubrics; cross-family judging is the remaining Case-B gap.
- **[04-llm-observability.md](04-llm-observability.md)** — *Case A, RICH.* The NDJSON event stream IS a live trace. `tool_call_start`/`tool_call_end{durationMs}` (`events.ts`) is a span; `durationMs` captured at the single `mcp.callTool` choke-point (`base.ts` L144–L149) times every span by construction; one captured `AgentEvent[]` serves the live UI (`useInvestigation` → `StatusLog`/`ReasoningTrace`/`TraceContent`, timestamped), the briefing's per-call `describeToolCall` status line, and event-for-event replay. Honest gaps: no Langfuse, no span aggregation, no queryable store, replay is verbatim-only. Exercise: persist an `ai_trace` table.
- **[05-regression-evals.md](05-regression-evals.md)** — *Case A.* The regression eval is its own pattern: **capture-then-score** with a two-mode comparator (structural diff for shape; LLM similarity judge for conclusion-level drift). The 30% semantic baseline vs 100% structural baseline is the load-bearing finding — outputs always have the right shape, but conclusions drift at a real rate on identical inputs. This is the rubric for "did my prompt edit make it worse."

## Reading order

Read in order: **04 (what observability the codebase HAS — the trace, the strength) → 01 (the eval set types, now real) → 02 (the scoring methods, all wired) → 03 (the judge biases — and the calibration-receipts discipline) → 05 (the regression eval as its own pattern).**

Start with 04 because it is the original anchored half — the live trace grounds the spans/durations/replay abstraction in real files.
