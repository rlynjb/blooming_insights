## Honest verdict on the page.tsx parent finding

**The parent finding stays documented-without-stub.** The 3 hook extractions ARE the action; the parent is the composite. After the three stubs ship, the page's body looks like this:

```tsx
const { mode, setMode, ready, switchMode } = useModePersistence();
const policy = useReconnectPolicy();
const {
  status, insights, workspace, coverage, traceItems,
  errorMessage, stepStatus, queryCount, demoSuffix,
} = useBriefingStream(mode, ready, policy.handle);
const { capturing, captureAll } = useDemoCapture(insights, workspace, traceItems);
const [activeQuery, setActiveQuery] = useState<string | null>(null);

return <main>...</main>;
```

That's the target structure — layout + composition, not orchestration. The page becomes "what's on screen, where state comes from, where actions go." No `useEffect`, no `readNdjson` call, no inline `switch(evt.type)`, no `sessionStorage` reads, no `fetch('/api/mcp/reset')`. The eight-concerns-at-one-altitude verdict becomes "layout component composing four hooks plus one local state slot for the active query."

**Emitting a parent-level stub here would be redundant.** It would say: "after the three hook stubs ship, delete the lifted code from `app/page.tsx`." That's the natural consequence of the three stubs landing; it doesn't need its own contract. The audit's call: keep the parent finding documented-without-stub; let the three hook stubs do the work. If the executor wants to verify the parent has actually collapsed, run `wc -l app/page.tsx` after each hook ships and confirm the LOC drops by the predicted ~340.

The `useModePersistence` warm-up (Hook A in the page-decomposition notebook) is the only piece left unaccounted-for. At ~10 LOC it's almost too small to be its own hook. **The audit's call: leave it inline, OR fold it into one of the three stubs as the executor's call.** Either is fine; the difference is one hook file vs zero. Both are documented in the parent finding.
