# Accessibility audit — blooming_insights — 2026-06-02

Descriptive snapshot of the a11y surface across the four `app/` routes and the 19 `components/` files. Per `audit-frontend-a11y.md`, this file describes what's there; it does not propose fixes, grade severity, or rank what should change.

Scope walked: `app/layout.tsx`, `app/page.tsx`, `app/investigate/[id]/page.tsx`, `app/investigate/[id]/recommend/page.tsx`, `app/debug/page.tsx`, and every `.tsx` in `components/{feed,investigation,chat,shared}`. The path mentioned in the task brief as `app/decide/[id]/page.tsx` does not exist; the decision flow lives at `app/investigate/[id]/recommend/page.tsx`.

The six lenses below are walked in spec order.

---

## Lens 1 — Keyboard

### Reachability

- All actionable controls are real `<button type="button">` or `next/link` `<Link>` elements (the elements browsers focus by default). No `<div onClick>` constructs were found across the surface.
  - feed mode toggle: `app/page.tsx:527-544`
  - clear-active-query: `app/page.tsx:580-595`
  - reconnect button: `app/page.tsx:663-687`
  - dev capture button: `app/page.tsx:717-738`
  - export markdown: `app/investigate/[id]/page.tsx:76-98`, `app/investigate/[id]/recommend/page.tsx:73-95`
  - back-link: `app/investigate/[id]/page.tsx:14-26`
  - "see recommendations" CTA: `app/investigate/[id]/page.tsx:175-193` (a `Link`)
  - debug action buttons: `app/debug/page.tsx:145-159, 194-235`
  - QueryBox input + submit: `components/chat/QueryBox.tsx:40-89`
  - "show reasoning" disclosure: `components/chat/StreamingResponse.tsx:249-263`
  - ToolCallBlock disclosure: `components/investigation/ToolCallBlock.tsx:37-86`
  - hypothesis disclosure: `<details><summary>` at `components/investigation/EvidencePanel.tsx:215-262`
  - InsightCard whole-card link: `components/feed/InsightCard.tsx:174-177` (a `Link` wraps the `<article>`)
  - CoverageGrid firing tiles: `components/feed/CoverageGrid.tsx:277-280` (a `Link` wraps the tile div)
  - recommendation deep-link: `components/investigation/RecommendationCard.tsx:220-237` (`<a target="_blank" rel="noopener noreferrer">`)

- The recommendation card's `prerequisites` row is a list of `<span>` items with no interactive handler — they are decorative status text, not actionable: `components/investigation/RecommendationCard.tsx:177-192`. The `✓` / `○` glyph beside each is wrapped in `aria-hidden`.

### Operability

- All keyboard activation flows through the platform `<button>` and `<a>`/`Link` semantics; no custom key handling competes with them. The single custom key handler is `Enter` on the QueryBox text input (`components/chat/QueryBox.tsx:45-50`), which calls `submit()` and respects the `disabled` flag.
- The CoverageGrid non-firing tile (`coverageState === 'unavailable'` or "clear") is a plain `<div>` and is not focusable. Only firing tiles become `Link`s.

### Tab order

- No `tabIndex` values are set anywhere in the scanned surface (grep returned nothing). DOM order matches visual order in the routes walked — header, then `ProcessStepper`, then `CoverageGrid`, then the feed cards, then the sticky `<aside>` log. On investigate pages: back-link, export button, header, `ProcessStepper`, `InvestigationSubject`, `EvidencePanel`, CTA, then the `<aside>` log.

### Traps

- No focus-trap or focus-restoration code exists. There is no modal/dialog/portal component in the scanned files, so there is no place a trap would normally be required. Disclosures (`<details>`, expanded buttons) leave focus on the trigger.

### Skip mechanisms

- No skip link is present. There is no `<nav>` element and no repeated navigation block above `<main>`, so on a fresh route load focus starts at the document and the first focusable target is whichever control sits at the top of `<main>` (on investigate pages, the back-link).

## Lens 2 — Focus

### Focus visibility

- No project CSS sets focus styles. `app/globals.css:1-81` declares color tokens, font setup, body styling, the `bi-fade-up`, `bi-progress`, and `bi-dots` animations (with `prefers-reduced-motion` overrides at lines 42 and 78-80), and a `border-radius: 4px` rule for `button, input, .card`. There is no `:focus`, `:focus-visible`, or `outline` rule.
- Tailwind v4 is imported at `app/globals.css:1` (`@import "tailwindcss"`); the project relies on user-agent default focus rings except where a component overrides them.
- One explicit removal: `components/chat/QueryBox.tsx:66` sets `outline: 'none'` inline on the text input and adds no replacement focus indicator.

### Focus on route change

- Navigation is via `next/link` (`InsightCard` → `/investigate/:id`, CoverageGrid firing tile → same, BackLink → `/`, "see recommendations" → `/investigate/:id/recommend`). No `useEffect` in the scanned files calls `.focus()` or moves focus on mount. Focus behaviour on route change is whatever Next.js' App Router default provides.

### Focus on disclosure open/close

- `ToolCallBlock` (`components/investigation/ToolCallBlock.tsx:37-86`) is a `<button aria-expanded={expanded}>` whose body conditionally renders below. Focus stays on the trigger button across toggle (the button is not removed from the DOM).
- `StreamingResponse`'s "show/hide reasoning" (`components/chat/StreamingResponse.tsx:249-263`) behaves the same: trigger button persists, body conditionally renders below.
- Hypothesis rows use native `<details>` / `<summary>` (`components/investigation/EvidencePanel.tsx:215-262`); browser-default focus handling applies. `summary` is styled with `listStyle: 'none'` (line 223) and `cursor` toggles between `'pointer'` and `'default'` depending on whether `h.reasoning?.trim()` is truthy. No disclosure marker is added back.

### Focus restoration

- Not implemented. The "clear active query" button (`app/page.tsx:580-595`) unmounts the `StreamingResponse` article; no code restores focus to wherever the query was submitted from. The reconnect button (`app/page.tsx:663-687`) triggers `window.location.href = '/'` — a full navigation that resets focus to the document.

## Lens 3 — Semantics

### Headings

- Section labels that look like headings ("anomaly coverage", "expected impact", "conclusion", "setup steps", "success metric", "prerequisites", "scope", "12-week trend", "funnel · this window vs prior", "investigating", "how this briefing was gathered", "how this was figured out", "how these were chosen") are `<div>` or `<span>` styled as small monospace text, not heading elements. - `ProcessStepper` step rows become a `<Link>` when an `href` is provided, otherwise a `<div>` (`components/shared/ProcessStepper.tsx:126-134`).

### Lists

- `EvidencePanel` "key evidence" uses `<ul>` with `listStyle: 'none'`: `components/investigation/EvidencePanel.tsx:273-285`.
- `RecommendationCard` "setup steps" uses an ordered `<ol>` (`components/investigation/RecommendationCard.tsx:200-207`).
- The bullet item for "key evidence" is preceded by a hand-drawn `<span aria-hidden>·</span>` glyph as the marker (`components/investigation/EvidencePanel.tsx:279-281`).
- The "prerequisites" row is a `<div>` with `<span>` children, not a list (`components/investigation/RecommendationCard.tsx:175-193`).
- The "hypotheses tested" rows are a `<div>` containing repeated `<details>` elements, not wrapped in a list (`components/investigation/EvidencePanel.tsx:212-263`).

- Insight cards on the feed are rendered with `.map` into sibling `<Link><article>` pairs inside a `<div>`, not a list (`app/page.tsx:704-710`).
- `ReasoningTrace` and the active-query response chain render items into sibling `<div>`s, not `<ol>` / `<ul>` (`components/investigation/ReasoningTrace.tsx:52-107`).

### Form structure

- `debug` page has two `<label>` elements, each immediately followed by its `<input>` / `<textarea>` (`app/debug/page.tsx:162-191`). No `htmlFor`/`id` association is used; the labels are sibling text, not programmatically tied to the controls.
- `QueryBox` text input has no `<label>` element. Its only label-equivalent is the `placeholder` attribute (`components/chat/QueryBox.tsx:51-55`), which changes copy when `disabled` is true. There is no `aria-label` on the input.
- No fieldsets, legends, radio groups, or checkbox groups appear in the scanned surface.
- The mode toggle (`app/page.tsx:526-544`) is a pair of `<button type="button">` siblings rather than radio inputs. The selected state is conveyed only by background/foreground colour swap; there is no `aria-pressed`, `aria-current`, or `role="radio"`.

### Tables

- No `<table>` is used. Tabular-looking data (the comparison rows on InsightCard, the funnel chip, the tile strips) is rendered with CSS grid / flex.
