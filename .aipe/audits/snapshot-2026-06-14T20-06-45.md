## Section 5 — Decisions

All 14 decisions from 06-02 still apply. Cleanup PR records four updates plus one new decision; these are captured here as state, not as deltas vs 19:53 — code didn't change.

**Updated — session-keyed in-memory state (was: deferred in 06-02).** Commit `c3d47a2`. The 6-line code comment at `lib/state/insights.ts:4-7` records the reasoning: "A single warm Vercel instance serves many users concurrently, so module-level Maps would bleed between sessions — and putInsights' clear() would wipe another user's feed mid-briefing." The fix preserves per-session `.clear()` (each briefing IS the current feed for that session) while closing the cross-session bleed.

**Updated — 30s per-tool MCP timeout (was: deferred in 06-02).** Commit `9597466`. `AbortSignal.timeout(TOOL_TIMEOUT_MS)` on both `SdkTransport.callTool` and `listTools`. Decision and tradeoff at `lib/mcp/transport.ts:21-30`: "Sibling of `retryCeilingMs: 20_000` in client.ts — that ceiling bounds a rate-limit retry wait, this one bounds the request itself." The retry ladder in `McpClient.callTool` only retries rate-limited *results*, not thrown errors, so a 30s timeout fails fast rather than burning another 30s wait inside the same route budget.

**Updated — tool-name allowlist on `POST /api/mcp/call` (was: deferred in 06-02).** Commit `39721e2`. A `Set<string>` built from `monitoringTools ∪ diagnosticTools ∪ recommendationTools ∪ bootstrapTools` (`app/api/mcp/call/route.ts:14-19`). Decision in code comment at `app/api/mcp/call/route.ts:11-13`: "Sourced from the same constants the agents use, so the boundary stays in sync with what the rest of the system already speaks." The /debug page was trimmed in the same commit to remove `whoami`; the test at `test/api/mcp-call-allowlist.test.ts:52-57` pins this end-to-end contract.

**New — server-side phase timing is the right grain for the 300s budget.** Commit `917d521`. `console.log({ route, sessionId, totalMs, phases })` in a `finally`, NOT a new event type on the NDJSON wire (`app/api/agent/route.ts:222-228, 307-318`). Decision in comment: "the summary still fires when a phase throws (the 300s-budget incident signal). Not on the NDJSON wire. Shape matches /api/briefing so a single Vercel filter reads both routes." The cost meter (`res.usage` at four call sites — commit `cc43e7d`) and the secret-redaction helper (commit `56e405b`) ride the same observability spirit: cheap logs that close blind spots without adding a metrics pipeline.

**Updated — error-response bodies carry message-only, not stack.** Commit `410e2bb`. Bodies in `/api/mcp/call`, `/api/mcp/tools`, `/api/mcp/tools/check`, `/api/mcp/capture` return `{ error: e.message }`. `console.error(e)` still fires so operators see the full stack in Vercel logs. Brings the four MCP routes into line with the streaming routes, which were already on the safer shape.

**Unchanged decisions (still load-bearing):** No database (RFC-001), NDJSON over `fetch` not SSE (RFC-002), deterministic supervisor in code not LLM (RFC-003), no LangChain/LlamaIndex, read-only tool whitelists, per-agent tool subsets, `McpClient.listTools` pass-through, `useInvestigation` no-cancel-on-cleanup, `bi_auth` 10-day maxAge no MFA, OAuth state CSRF re-check disabled (`consumeState` defined at `lib/mcp/auth.ts:230-235`, unwired), `minIntervalMs = 1100`, tool-result truncation budgets (`MAX_TOOL_RESULT_CHARS = 16_000` for the model, `TRUNC = 4000` for the UI), Vercel Pro non-negotiable, `AGENT_MODEL = 'claude-sonnet-4-6'` uniform.
