## Per fix-now template-by-shape choice (the v1.67.1 delta)

The new orchestrator's shape rubric:
- **logic only** (no UI surface touched) → `cleanup-<name>.md` using `refactor.md` template (API stays identical, no behaviour change)
- **UI behaviour** (state / effects / components / data flow / event semantics / a11y) → `cleanup-frontend-<name>.md` using `refactor-frontend-behaviour.md` template (extends logic + pixels stay identical, event order identical, a11y identical, framework context named)
- **visual surface only** (CSS / design tokens / semantic HTML) → `cleanup-visual-<name>.md` using `refactor-frontend-visual.md`

```

  ─────────     ───────────────────────────────────  ─────────       ──────────────────────────────

  3             NDJSON-line reader extraction        UI behaviour    refactor-frontend-behaviour.md
                (4 sites — 3 are React components)   (effects +      → cleanup-frontend-<name>.md

```

**Why #3 is frontend-behaviour, not logic-only.** The four NDJSON sites are *not* all server-side. Three of the four are React-component effects with mount-cycle semantics:
- `app/page.tsx:323-464` — main briefing effect; `cancelOn: () => !mounted` semantics
- `lib/hooks/useInvestigation.ts:184-208` — client hook, deliberately doesn't cancel on cleanup (StrictMode reason — see `lib/hooks/useInvestigation.ts:31-36`)
- `components/chat/StreamingResponse.tsx:107-126` — React component effect

The fourth (`app/page.tsx:181-203` capture path) is button-handler code inside a React component. Every consumer is in a framework lifecycle. The kernel itself is pure (just stream→bytes→lines→events), but the *call sites that change* are React effects and event handlers — which means `refactor-frontend-behaviour.md`'s must-not-change list is the load-bearing one: visible UI behaviour, event semantics, network/storage behaviour, accessibility. The frontend-behaviour template also forces the spec to name **framework context** (React 19, Next.js 16 App Router, `reactStrictMode` on) which is exactly the thing that breaks if the refactor silently changes cleanup semantics.

The morning's stub (`cleanup-extract-ndjson-reader.md`, written under v1.66.13's single-template world) is technically usable — but it doesn't carry the must-not-change list that frontend-behaviour requires (pixels-identical, event-order-identical, a11y-identical). Under v1.67.1, the correct stub name would be `cleanup-frontend-extract-ndjson-reader.md` using `refactor-frontend-behaviour.md`'s richer constraint list.

**Per the hard rules, this audit does not overwrite the existing stub.** The delta is noted here; if/when the refactor is executed, the executor should fill in the must-not-change list using the frontend-behaviour template's expanded checklist rather than the general `refactor.md` checklist.
