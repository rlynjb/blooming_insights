# Snapshot — 2026-06-02

> **What blooming insights IS today.** A Next.js 16 + React 19 + TypeScript single-workspace AI analyst that hangs four Claude agents (monitoring, diagnostic, recommendation, query) off one Bloomreach loomi connect MCP server, with a shared `runAgentLoop` (`lib/agents/base.ts:48-176`), a hand-built `McpClient` (`lib/mcp/client.ts:79-172`) that owns TTL cache + 1.1s spacing + bounded exp-backoff retry + parsed `Retry-After`, NDJSON `AgentEvent` streams (`lib/mcp/events.ts:4-17`) that every layer speaks, an encrypted-cookie OAuth boundary (`lib/mcp/auth.ts:51-79`) that replaces a database, and 169 vitest tests across 18 files scoped to `lib/`. ~3,400 LOC across 58 source files in `app/`, `components/`, and `lib/`. Built for the Loomi Connect AI Hackathon (Track 3, deadline 2026-06-02), now also serving as a portfolio artifact for a frontend-to-AI-engineering pivot. No database, no LLM eval harness, no metrics aggregator, no Sentry, no circuit breaker, no real-customer validation.

---

## Section 1 — Identity

**One-paragraph description.** blooming insights watches a Bloomreach Engagement ecommerce workspace (project `wobbly-ukulele`) and answers three questions for non-analyst operators: *what changed, why, and what should I do.* It runs the loop a human data analyst runs — monitoring detects significant period-over-period changes in metrics (revenue, funnel, traffic), diagnostic generates and tests hypotheses against the data, recommendation maps the diagnosis to concrete Bloomreach features (scenario / segment / campaign / voucher / experiment). The agents' reasoning trace is the product surface, not a debug log — every conclusion ships with its tool calls, EQL queries, current-vs-prior numbers, and a streamed thought log (`.aipe/project/context.md` L3-L21).

**Primary user.**
- Named in spec (`blooming-insights-spec.md:55`): "merch leads, store operators without analysts."
- Actual audience the build was optimized for: contest judges (Loomi Connect AI Hackathon, Track 3) and hiring panels reviewing the portfolio (`rehearse-problem-selection/01-problem-brief.md:69-93`).
- No real merchant has touched, validated, or asked for this product (`rehearse-problem-selection/01-problem-brief.md:88-92`).

**Stage.** **Contest-deadline MVP / portfolio artifact.** Build window opened 2026-05-26; submission deadline 2026-06-02 4:00 pm pst (`blooming-insights-spec.md:41-43`). Not in production, no real users, no SLA. Active in the sense that the most recent commits (`a32d4dc`, `e9c3f99`) are documentation passes for the `aipe` toolkit, not product features — the product code itself is feature-frozen for the contest.

**Persistence layer.** **No database.** State lives in eight tiers (`study-system-design/audit.md:87-101`):
1. In-memory `Map`s (insights, investigations, schema, McpClient cache) — instance lifetime.
2. `AsyncLocalStorage` request store (`lib/mcp/auth.ts:46-47, 86-104`) — per request.
3. `sessionStorage` (per tab) — cross-instance carrier for live-mode insights.
4. `localStorage` (`bi:mode` only).
5. Cookies — `bi_session` (httpOnly UUID, session) + `bi_auth` (AES-256-GCM, 10 days). **The only production storage that survives instance recycle and carries application state.**
6. Filesystem (dev only, gated by `NODE_ENV === 'development'`) — `.auth-cache.json`, `.investigation-cache.json`.
7. Committed JSON (per deploy) — `lib/state/demo-insights.json`, `lib/state/demo-investigations.json`.
8. Bloomreach upstream — the system-of-record, read-only by construction.

**Deploy target.** Vercel Pro (the 300s `maxDuration` requires Pro; Hobby's 60s cap won't fit a typical ~100-115s investigation — comment at `app/api/agent/route.ts:18-19`).

**Anything unusual or non-default.**
- AES-256-GCM-encrypted OAuth-state cookie as the only durable production state (`lib/mcp/auth.ts:51-79`) — replaces a database for the session boundary.
- `AsyncLocalStorage`-scoped per-request cookie store (`lib/mcp/auth.ts:46-47, 86-104`) — exists because Next's request/response cookie split breaks read-after-write.
- NDJSON over `fetch` + `ReadableStream`, not SSE / `EventSource` (`lib/mcp/events.ts:4-17`, `app/api/agent/route.ts:167-264`).
- One file (`lib/agents/base.ts:48-176`) implements the ReAct loop for all four agents; no LangChain, no LlamaIndex, no framework.
- Schema-gated coverage (`lib/agents/categories.ts:1-160`) inspects workspace schema before the monitoring agent runs, so the model never spends budget on categories the data can't support.

## Section 3 — Architecture

**Top-level structure.** Five bands, flat (`study-system-design/audit.md:9`).

```
  app/                       (UI + route handlers, Next.js App Router)
    page.tsx                       feed page (817 LOC)
    investigate/[id]/page.tsx      diagnose page (225 LOC)
    investigate/[id]/recommend/    recommend page (197 LOC)
    api/briefing/                  monitoring scan (265 LOC)
    api/agent/                     investigation step + query (268 LOC)
    api/mcp/{callback,reset,call,tools,tools/check,capture,capture-demo}/
  components/                 (React UI)
    feed/ investigation/ chat/ shared/
  lib/
    agents/                  4 agents + shared runAgentLoop
    mcp/                     McpClient + transport + auth + schema + validate + events
    state/                   insights Map + investigations replay store
    hooks/                   useInvestigation (216 LOC NDJSON-parser hook)
    insights/                derived-field calculators
    export/                  markdown export
  test/                      18 files / 169 tests, all in lib/ surface
```

**State model.** Eleven pieces of state across seven owners, all enumerated in `study-system-design/audit.md:46-60`. Lifetime ladder: `useState` per mount → `sessionStorage` per tab → cookies (10 days for `bi_auth`) → in-memory `Map`s per Vercel instance → committed JSON per deploy → Bloomreach (the only true source of truth). **Nothing in-process survives an instance recycle in production except the encrypted `bi_auth` cookie and committed demo JSON** (`study-system-design/audit.md:61-63`).

**Data flow.** Three live flows + one replay shortcut, all on the same NDJSON `AgentEvent` wire format (`study-system-design/audit.md:26-32`):
- Flow 1 — briefing: `/api/briefing` runs schema bootstrap → coverage gate → monitoring scan → emits `workspace`, `coverage_item ×10`, `tool_call_*`, `insight ×N`, `done`. ~30-60s cold start.
- Flow 2 — investigation step: `/api/agent?step=diagnose|recommend|null`. Cache-replay shortcut at top (`app/api/agent/route.ts:127-141`) replays cached events at 180ms ticks. Live path runs DiagnosticAgent or RecommendationAgent (≤6 tool calls), 30-90s typical.
- Flow 3 — query: `/api/agent?q=`. `classifyIntent` → `QueryAgent.answer` → plain text. Never cached.

The route writes the first `reasoning_step` event INSIDE the stream (`app/api/agent/route.ts:196-203`) so the browser sees activity in <100ms before the 4-call MCP bootstrap finishes — perceived latency trick (`study-performance-engineering/audit.md:87-89`).

**External integrations.**
- Bloomreach loomi connect MCP (alpha, `https://loomi-mcp-alpha.bloomreach.com/mcp`) — wrapped by `McpClient` (`lib/mcp/client.ts`) over `McpTransport` (`lib/mcp/transport.ts`) over the MCP SDK, with `BloomreachAuthProvider` (`lib/mcp/auth.ts:160-218`) underneath. OAuth 2.0 + PKCE + Dynamic Client Registration. Rate limit ~1 req/s/user GLOBAL; alpha server revokes tokens after minutes (`.aipe/project/context.md:88-89`).
- Anthropic SDK — 4 `messages.create` call sites: `lib/agents/base.ts:102`, `lib/agents/diagnostic.ts:97`, `lib/agents/recommendation.ts:96`, `lib/agents/intent.ts:18`.

**Boundaries between layers.** Three real boundaries, one cosmetic (`study-system-design/audit.md:11-17`):
- **B1 browser ↔ route** — trust flips HOSTILE → OURS. Enforced by httpOnly `bi_session` UUID + AES-256-GCM `bi_auth` cookie.
- **B2 McpClient ↔ Bloomreach** — trust flips OURS → UPSTREAM + rate flips to ~1 req/s/user.
- **B3 model output ↔ typed value** — trust flips UNTRUSTED text → TYPED value via `parseAgentJson` + per-shape type guards in `lib/mcp/validate.ts`; mismatch → `FALLBACK`.
- Cosmetic: route ↔ agent loop — same process, same `Map`s, same heap.

## Section 4 — Features

Ordered by maturity, most stable first.

**1. NDJSON `AgentEvent` streaming + cache replay (stable).** The 8-line discriminated union in `lib/mcp/events.ts:4-12` is the contract every layer speaks. Live runs and cached replays emit identical streams; replay paces at 180ms per event so the UI animates the same. Used by `/api/briefing`, `/api/agent`, the `useInvestigation` hook, the `StatusLog` component. `Cache-Control: no-cache, no-transform` prevents intermediary buffering (`study-performance-engineering/audit.md:91`).

**2. `McpClient` provider abstraction (stable).** `lib/mcp/client.ts:79-172`. 3-method surface (`callTool`, `listTools`, `close`); hides TTL cache (60s, exact-match keyed on `${name}:${JSON.stringify(args)}`), spacing gate (`minIntervalMs = 1100`), exponential-backoff retry (`maxRetries = 3`, parsed `Retry-After` from Bloomreach's prose error format, `retryCeilingMs = 20s`), error tagging via `McpToolError`, no-cache-on-error invariant. 14 tests in `test/mcp/client.test.ts` (`study-software-design/audit.md:25-26`).

**3. Encrypted-cookie OAuth boundary (stable).** `lib/mcp/auth.ts:33-143, 160-218`. Three OAuth storage backends selected on `NODE_ENV` (file in dev, memory in test, AES-256-GCM-encrypted cookie in prod). The `bi_auth` cookie is the only durable production state. `BloomreachAuthProvider` implements the MCP SDK's `OAuthClientProvider` interface with PKCE + DCR. 14 tests in `test/mcp/auth.test.ts`. RFC documented at `.aipe/rehearse-design-doc/01-no-database-encrypted-cookie.md`.

**4. `runAgentLoop` shared ReAct kernel (stable).** `lib/agents/base.ts:48-176`. One function, four callers (`MonitoringAgent`, `DiagnosticAgent`, `RecommendationAgent`, `QueryAgent`). Hides Anthropic tool-use protocol, `tool_use` block extraction, `tool_result` feedback, the `forceFinal` tool-less synthesis turn pattern (`lib/agents/base.ts:88-101`), tool-call budget enforcement (`maxToolCalls`), graceful turn-budget termination. 8 tests in `test/agents/base.test.ts`.

**5. Four-agent sequential pipeline (stable).** Monitoring (6 tool calls, schema-gated coverage, output `Anomaly[]`) → Diagnostic (6 calls, output `Diagnosis`, with `synthesize()` recovery in `lib/agents/diagnostic.ts:87-126`) → Recommendation (4 calls, output `Recommendation[]`, same `synthesize()` recovery in `lib/agents/recommendation.ts:82-132`) → QueryAgent runs separately on `?q=`. Deterministic supervisor — code, not an LLM router (RFC at `.aipe/rehearse-design-doc/03-deterministic-supervisor-not-llm-router.md`).

**6. Type-guard trust boundary (stable).** `lib/mcp/validate.ts:1-57`. `parseAgentJson` (fenced → bare → substring-scan fallback chain) + per-shape guards (`isAnomalyArray`, `isDiagnosis`, `isRecommendationArray`) + per-agent `FALLBACK` constants. 25 tests in `test/mcp/validate.test.ts` using the "acceptance + per-gate rejection" discipline (`study-testing/audit.md:80-83`).

**7. Schema-gated coverage (stable).** `lib/agents/categories.ts:1-160`. Inspects workspace schema shape via `coverageFor` / `coverageReport` before monitoring runs, so the agent never spends budget on categories the data can't support. 12 tests in `test/agents/categories.test.ts`. Emits `coverage_item` events into the stream.

**8. Intent router (stable).** `lib/agents/intent.ts:1-31`. Heuristic + `claude-haiku-4-5` classifier in front of QueryAgent. Routes `?q=` to QueryAgent vs the structured agent pipeline.

**9. Demo vs live mode (stable).** `?demo=cached` serves the committed snapshot (`lib/state/demo-insights.json` + `demo-investigations.json`) as plain JSON; investigations replay committed events filtered per step. Live mode runs the agents against Bloomreach. The 180ms-per-event replay (`app/api/agent/route.ts:127-141`) makes cached indistinguishable from live in the UI. Built because the alpha MCP server is too fragile for a live presentation.

**10. OAuth 2.0 + PKCE + DCR flow (stable, untested).** Wired via `BloomreachAuthProvider` (`lib/mcp/auth.ts:160-218`) + `connectMcp` / `completeAuth` (`lib/mcp/connect.ts`). The crypto round-trip is tested (`test/mcp/auth.test.ts`); the end-to-end OAuth orchestration has zero tests (`study-testing/audit.md:23` — Gap B).

**11. Auto-reconnect on `invalid_token` (stable).** `app/page.tsx:410-427`. `bi:reconnecting` sessionStorage flag prevents infinite loops. Handles alpha-server token revocation.

**12. Dev-only one-click demo capture (stable).** `/api/mcp/capture-demo` runs the live briefing + each investigation and writes `lib/state/demo-*.json`. Used to refresh the demo snapshot before a presentation.

**13. Markdown export (stable).** `lib/export/investigationMarkdown.ts` (75 LOC). Used by both investigate pages.

**14. UI surface — feed + two investigate pages + QueryBox (rough at the seams).** `app/page.tsx` (817 LOC, 14 `useState` slots, 8 concerns in one component — `study-software-design/audit.md:29`), `components/feed/InsightCard.tsx` (495 LOC, ~150 inline-style objects mixed with Tailwind — `study-software-design/audit.md:12, 97`), `lib/hooks/useInvestigation.ts` (216 LOC NDJSON parser in a hook). Functional and shipped; the named "rough" lives here, not in the agent or MCP layers.

## Section 5 — Decisions

The reasoning behind choices not obvious from the code. Three of these are documented as RFCs in `.aipe/rehearse-design-doc/`.

**No database (RFC-001, `.aipe/rehearse-design-doc/01-no-database-encrypted-cookie.md`).** State is held in `Map`s + an AES-256-GCM-encrypted cookie + committed demo JSON. Reasoning: the system-of-record IS Bloomreach (every insight is a transformed Bloomreach query result), the contest window was 7 days, and serverless functions are read-only FS in production. The encrypted cookie is what makes OAuth survive instance hops across Vercel's ephemeral functions. **Rejected alternatives:** Postgres (overhead for state with no relational features today), Vercel KV / Upstash Redis (would be the smallest viable add if shared feeds become a requirement, ~$5/month).

**NDJSON over `fetch` + `ReadableStream`, not SSE (RFC-002, `.aipe/rehearse-design-doc/02-ndjson-over-fetch-not-sse.md`).** Reasoning: NDJSON's wire format is a typed discriminated union the producer and consumer share via `lib/mcp/events.ts`; SSE's `event:` / `data:` framing would have required a parallel typed contract. **Rejected:** SSE / `EventSource` (would lose the `AbortController` + `fetch` body-reader symmetry between live and replay paths).

**Deterministic supervisor in code, not an LLM supervisor (RFC-003, `.aipe/rehearse-design-doc/03-deterministic-supervisor-not-llm-router.md`).** The route handler enforces pipeline order; agents don't choose what runs next. Reasoning: the pipeline is fixed (`monitoring → diagnostic → recommendation`), the cost of an LLM router decision (one more `messages.create`, more latency, more failure modes) doesn't earn its keep when the order is known. **Rejected:** an LLM-based orchestrator (introduces non-determinism, harder to test, costs tokens per step).

**No LangChain / LlamaIndex / agent framework.** Reasoning: the agent loop is 128 lines (`lib/agents/base.ts:48-176`) and reused across 4 agents with zero duplication. A framework would replace a known small surface with an unknown large one. Tradeoff: every framework feature (memory, planners, etc.) has to be re-derived if needed.

**Read-only tool whitelists by name pattern (`lib/mcp/tools.ts:5-40`).** All `list_*`, `get_*`, `execute_analytics*` — no `create_*`, `update_*`, `delete_*`. Reasoning: capability minimization is the structural defense against prompt injection, not input validation (`study-security/audit.md:69`). Even if a prompt is injected, the agent has no capability to write.

**Per-agent tool subsets, not the union.** `MonitoringAgent` gets 13 read tools, `DiagnosticAgent` 18 (includes `list_customers`), `RecommendationAgent` 10, `QueryAgent` the full union (~30). Reasoning: tighter capability for the structured agents, widest for the free-form query path. Tradeoff named: `QueryAgent` pairs the widest capability with no output gate (`study-security/audit.md:72-73`).

**`McpClient.listTools` is a literal pass-through (`lib/mcp/client.ts:168-171`).** Justified by an inline comment: the tool list is read once at startup, doesn't count against the rate limit, and failures are fatal anyway (`study-software-design/audit.md:59`). The only pass-through in the codebase.

**`useInvestigation` deliberately doesn't cancel its fetch on cleanup (`lib/hooks/useInvestigation.ts:31-36`).** Reasoning: React StrictMode's mount-clean-remount aborted streams and emptied logs; the started-guard prevents double-fetch. Cost: a mid-investigation navigate-away wastes Anthropic + MCP budget.

**`bi_auth` cookie maxAge = 10 days, no MFA.** Reasoning: single-user demo shape; cookie theft = full impersonation for 10 days is acceptable for the contest, would block a B2B SaaS ship (`study-security/audit.md:24`).

**OAuth state CSRF re-check disabled in callback (`app/api/mcp/callback/route.ts:22-26`).** Reasoning: the MCP SDK calls `state()` multiple times per flow; naive store-last-compare-once rejects legitimate callbacks. `consumeState` is implemented and tested in `lib/mcp/auth.ts:230-235` but not wired. Accepted risk on the assumption the SDK handles it internally (`study-security/audit.md:22`).

**`minIntervalMs = 1100`, not 1000 (`lib/mcp/connect.ts:82-92`).** 6-line comment above the literal carries the budget math: Bloomreach's "1 per N second" prose error format implies the next-call window opens at N+ε; 1100ms gives ε = 100ms of safety. The comment IS the design (`study-software-design/audit.md:95`).

**Capture `tool_result` truncated to 16KB for the model, 4KB for the UI stream.** `MAX_TOOL_RESULT_CHARS = 16_000` (`lib/agents/base.ts:29`), `TRUNC = 4000` (`app/api/agent/route.ts:99-103`). Same pattern, two consumers, two budgets. Reasoning: the model needs enough context to reason; the UI needs enough to render the "what was gathered" panel without bloating the response body.

**Vercel Pro is non-negotiable.** The 300s `maxDuration` (`app/api/agent/route.ts:20`) is pinned at Vercel Pro's max; Hobby's 60s won't fit a ~100-115s investigation (comment at `app/api/agent/route.ts:18-19`).

**`AGENT_MODEL = 'claude-sonnet-4-6'` used uniformly for the 4 main agents** (`lib/agents/base.ts:9`); `claude-haiku-4-5-20251001` for intent classification only. No decision artifact about why sonnet vs opus; no cost/quality tradeoff measured (`recon-2026-06-02.md:56`).

### Deferred with intent

**No database.** Explicit "no" decision (RFC-001). System-of-record is Bloomreach. Smallest viable add named (Vercel KV ~$5/month, ~200 LOC) for the day shared feeds become a requirement.

**No LLM eval harness.** No `evals/` directory at repo root (confirmed by `find`). No goldset, no judge, no agreement rate, no regression set. The substrate IS built (NDJSON trace + `durationMs` + `lib/state/investigations.ts` replay would BE the harness if wired) — the harness itself is the gap (`study-testing/audit.md:140`, `study-ai-engineering/05-evals-and-observability/README.md:7`).

**No circuit breaker on `McpClient`.** `lib/mcp/client.ts:121-132` — retry ladder works per-call, no consecutive-failure tracking, no fail-fast during cooldown (`study-system-design/audit.md:120`). During a 5-minute Bloomreach outage, every concurrent request burns 3 retries × ~12s = ~36s before failing. Recognized in the audits, unshipped.

**No `res.usage` cost meter.** 4 `anthropic.messages.create` call sites, all drop `res.usage` on the floor (`lib/agents/base.ts:102`, `lib/agents/diagnostic.ts:97`, `lib/agents/recommendation.ts:96`, `lib/agents/intent.ts:18`). No per-investigation cost number anywhere (`study-performance-engineering/audit.md:22, 111`).

**No metrics aggregator.** `durationMs` measured per call (`lib/mcp/client.ts:112,134`), nothing rolls it up. No `lib/metrics.ts`, no histogram, no `/api/metrics` endpoint (`study-debugging-observability/audit.md:43-47, 101`).

**No structured logger.** 4 `console.error` calls total across the backend (`app/api/agent/route.ts:160,256`, `app/api/briefing/route.ts:166,248`). No `lib/log.ts`, no levels, no correlation IDs, no redaction (`study-debugging-observability/audit.md:37-41`).

**No prompt versioning, no A/B, no eval score.** Prompts in `lib/agents/*.ts` and `lib/agents/prompts/*.md` are hand-edited; every edit ships with zero quality signal (`recon-2026-06-02.md:57`).

**No customer research.** No interviews, no waitlist, no design-partner pilots, no support-ticket analysis. Zero customer-discovery artifacts in the repo (`rehearse-problem-selection/01-problem-brief.md:175-181`).

**No multi-tenant.** Single workspace `wobbly-ukulele` only. Scope cut driven by the 7-day contest window (`blooming-insights-spec.md:33`, `rehearse-problem-selection/01-problem-brief.md:281-290`).

**No write-path actions.** Read-only MCP by construction (`lib/mcp/tools.ts:5-40`); recommendations are suggestions only. Spec-stated and architecturally enforced (`blooming-insights-spec.md:32`).

**No Sentry / error tracker / on-call rotation / runbooks / SLOs / PagerDuty.** Detect layer for prod is "the user reports it" or "the developer reads Vercel logs." Rationally deferred for solo demo (`study-debugging-observability/audit.md:71-73`).

**No backend trace sink (OpenTelemetry / Langfuse / Datadog).** The `AgentEvent` stream is span-shaped already (`tool_call_start { toolName, agent }` → `tool_call_end { durationMs, result?, error? }`) but lives only in the UI and the in-memory cache (`study-debugging-observability/audit.md:21`).

**No CSRF token on POST routes, no Origin check on GETs with side effects.** `/api/briefing` and `/api/agent` both spend Anthropic tokens via GET (`study-security/audit.md:20`). `POST /api/mcp/reset` is trivially CSRF-able.

**No CI security gates.** No Dependabot/Renovate, no `npm audit` step in CI, no SBOM (`study-security/audit.md:63`).

**No route-layer or UI tests.** `app/api/*` has 0 tests; `components/` has 0 tests; `app/*/page.tsx` has 0 tests (`study-testing/audit.md:9-18`). `vitest.config.ts` is `environment: 'node'` with no `@testing-library/react` dep (Gap C, acceptable for scope).

**No Playwright / e2e smoke.** No `test/api/` directory, no end-to-end NDJSON stream test (`study-testing/audit.md:33, 125`).

**No prompt-prefix caching (Anthropic feature).** `lib/agents/base.ts:92-101` has no `cache_control` blocks; the 5-10KB system prompts re-tokenize every turn (`study-performance-engineering/audit.md:116`).

**No load testing, no profiler integration, no Web Vitals.** Stated `not yet exercised` (`study-performance-engineering/audit.md:25-30, 96`).

## Section 7 — Risks and rough edges

Observation only.

**The `lib/state/insights.ts:4` global `Map`.** One *global* `Map<id, Insight>` per Vercel instance; `putInsights()` at L30 calls `insights.clear()` (L36) — correct for one user, wipes another user's data when two land on the same warm instance (`study-system-design/audit.md:132, 150`). Named in 4 separate study guides; ~30 LOC fix; unshipped. Active at ~10 concurrent users.

**The `app/page.tsx` 817-LOC component.** 14 `useState` slots, 8 concerns at one altitude (rendering, NDJSON stream parsing, reconnect policy, demo capture, mode toggling, coverage accumulation, trace accumulation, stepper-state derivation). The worst shallow module in the repo, fires #1 on the design-red-flags audit (`study-software-design/audit.md:107, 136`).
