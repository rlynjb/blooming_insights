# Model-output type guards

## Subtitle

Runtime type-guarded parsing at the untrusted-input boundary · Industry standard (parse-don't-validate), Project-specific implementation (`parseAgentJson` + `isAnomalyArray` / `isDiagnosis` / `isRecommendationArray`)

---

## Zoom out — where this concept lives

The Anthropic API is a trusted counterparty in one direction only. Your request goes over TLS with an API key; the response comes back over the same channel. But the *content* of the response — the model's text — is the least-controlled input in the system. It was generated by an LLM, possibly influenced by prompt injection in the tool results, possibly hallucinated, possibly out of date with your Zod-less type contract.

Every claim the model makes has to prove its shape before your code takes it seriously.

```
  Zoom out — where the type guards sit

  ┌─ Service layer ────────────────────────────────────────┐
  │                                                          │
  │   ┌ Agent loop ──────────────────────────────────────┐  │
  │   │  anthropic.messages.create(...) → response         │  │
  │   │  response.content → text                           │  │
  │   └───────────────────────┬──────────────────────────┘  │
  │                            │                              │
  │                            │  text (untrusted!)           │
  │                            ▼                              │
  │   ┌ lib/mcp/validate.ts ─────────────────────────────┐  │
  │   │  parseAgentJson(text)     — extract & parse JSON  │ ★│
  │   │  isAnomalyArray(v)        — Anomaly[] shape       │  │
  │   │  isDiagnosis(v)           — Diagnosis shape       │  │
  │   │  isRecommendationArray(v) — Recommendation[]      │  │
  │   └───────────────────────┬──────────────────────────┘  │
  │                            │  narrowed to real types     │
  │                            ▼                              │
  │   ┌ State + UI ──────────────────────────────────────┐  │
  │   │  putInsights, saveInvestigation, NDJSON stream    │  │
  │   └───────────────────────────────────────────────────┘  │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
```

Everything after the type-guard boundary can assume real `Anomaly` / `Diagnosis` / `Recommendation` values. Everything before it is a `string` that could be anything.

## Structure pass — layers, axis, seams

**Layers.** Anthropic response → text → JSON parse → type guard → domain object → downstream sinks.

**Axis: trust.** What's untrusted at each layer?

- Response envelope: the shape (role, content, usage) is trustworthy — that's the SDK's guarantee.
- Text content: not trustworthy — the model wrote it.
- Parsed JSON (unknown): shape unclear — could be anything JSON-parses.
- Guarded value (`Anomaly[]`, `Diagnosis`, etc.): typed and shape-verified — safe to hand downstream.

**Seam.** One boundary carries all the weight: the transition from `unknown` to a typed value. Two guards work together:

1. **Parse guard** — `parseAgentJson` throws if the text isn't JSON at all. Fenced code blocks are unwrapped; substring scans catch prose-wrapped JSON.
2. **Shape guard** — `isAnomalyArray` etc. narrow `unknown` to a specific type by checking every load-bearing field. Returns `false` on any mismatch; caller decides fallback.

Hand off.

---

## How it works

### Move 1 — the mental model

You know the "parse, don't validate" idea (Alexis King)? "Validate" gives you a boolean and leaves the type wide; "parse" gives you a narrower type or fails. In TypeScript this becomes user-defined type predicates: `function isX(v: unknown): v is X`. The compiler now knows `v` is `X` inside the guarded branch.

The pattern's shape:

```
  Parse-don't-validate — the pattern

  unknown ────► parse ────► unknown (JSON) ────► guard ────► X
                  │                                │
                  │ throws on non-JSON            │ returns false
                  ▼                                ▼
             caller catches                  caller falls back
             (fallback path)                 (fallback path)
```

The load-bearing part: the guard checks every field that downstream code will *access*, not just a couple. A guard that only checks `metric: string` and lets `severity: "urgent"` through breaks downstream when the UI expects `severity ∈ {critical, warning, info, positive}`.

### Move 2 — walkthrough

**The parse gate.** `parseAgentJson` handles three formats the model might emit: raw JSON, JSON inside a ```` ```json ```` fence, or JSON preceded by prose ("Here are the anomalies:\n[...]"). All three land at `JSON.parse`.

**File:** `lib/mcp/validate.ts`
**Function:** `parseAgentJson`
**Line range:** 3-13

```ts
export function parseAgentJson(text: string): unknown {
  const fence = text.match(/```(?:json)?\s*([\s\S]*?)```/i);
  const candidate = (fence ? fence[1] : text).trim();
  try { return JSON.parse(candidate); } catch { /* fall through to substring scan */ }
  const start = candidate.search(/[[{]/);
  const end = Math.max(candidate.lastIndexOf(']'), candidate.lastIndexOf('}'));
  if (start >= 0 && end > start) {
    return JSON.parse(candidate.slice(start, end + 1));
  }
  throw new Error('no parseable json in agent output');
}
```

Reading line by line:

- Line 4 — regex tries a fenced block first. `/```(?:json)?\s*([\s\S]*?)```/i` matches ```` ```json ... ``` ```` or ```` ``` ... ``` ````.
- Line 5 — `candidate` is either the fence contents or the full text; `.trim()` handles leading whitespace.
- Line 6 — try direct parse. If the model returned clean JSON, this wins.
- Lines 8-11 — fallback substring scan: find the first `[` or `{`, and the last `]` or `}`, and parse the substring between them. Handles "Here are the anomalies:\n[{...}, {...}]\n\nThat's the summary."
- Line 12 — nothing parseable found. Throw. Callers catch and fall back to an empty result.

What breaks if you use `JSON.parse` directly without the fence unwrap: models frequently emit fenced JSON when told "reply in JSON format." Every response fails to parse. The fence-unwrap is not defense-in-depth; it's the primary path.

What the substring scan doesn't defend against: JSON with a valid opening `[` but truncated closing bracket. `JSON.parse` will throw and the outer `parseAgentJson` will throw. Which is correct — corrupted JSON should fail loudly, not be salvaged.

**The shape guards.** Three functions, all shaped `(v: unknown): v is T`.

**File:** `lib/mcp/validate.ts`
**Function:** `isAnomalyArray`
**Line range:** 17-27

```ts
const SEVERITIES: Severity[] = ['critical', 'warning', 'info', 'positive'];

export function isAnomalyArray(v: unknown): v is Anomaly[] {
  return Array.isArray(v) && v.every((a) =>
    !!a && typeof a === 'object' &&
    typeof (a as any).metric === 'string' &&
    Array.isArray((a as any).scope) &&
    !!(a as any).change && typeof (a as any).change.value === 'number' &&
    ((a as any).change.direction === 'up' || (a as any).change.direction === 'down') &&
    typeof (a as any).change.baseline === 'string' &&
    SEVERITIES.includes((a as any).severity)
  );
}
```

Every field is checked. Not just presence — value-space too. `severity` must be one of four literal strings; `change.direction` must be `'up'` or `'down'`. The trust assumption named: any anomaly that passes this guard is safe to hand to `anomalyToInsight` (`lib/state/insights.ts:25`) which will then call `deriveInsightFields` and populate the UI feed.

**File:** `lib/mcp/validate.ts`
**Function:** `isDiagnosis`
**Line range:** 29-35

```ts
export function isDiagnosis(v: unknown): v is Diagnosis {
  if (!v || typeof v !== 'object') return false;
  const d = v as any;
  return typeof d.conclusion === 'string'
    && Array.isArray(d.evidence)
    && Array.isArray(d.hypothesesConsidered);
}
```

Weaker than `isAnomalyArray`. Only three fields checked, and the arrays' *contents* aren't validated. What that lets through: a diagnosis with `evidence: [null, undefined, 42]` passes. The downstream code that reads `evidence` — `components/investigation/EvidencePanel.tsx` — would then need to defensively render each item.

The tradeoff (not the audit's job to fix, but worth naming): tight guards get expensive when the shape is nested. Loose guards push validation cost to the render layer. The current shape says "diagnoses are common but their fields are all optional-ish; the UI already handles missing pieces." Acceptable, and worth revisiting if a component ever crashes on a malformed evidence item.

**File:** `lib/mcp/validate.ts`
**Function:** `isRecommendationArray`
**Line range:** 42-57

```ts
const FEATURES = ['scenario', 'segment', 'campaign', 'voucher', 'experiment'];
const CONFIDENCE = ['high', 'medium', 'low'];

// The agent emits recommendations WITHOUT an `id` (the system assigns ids after
// validation), so we validate the array of the id-less shape.
export function isRecommendationArray(v: unknown): v is Omit<Recommendation, 'id'>[] {
  return Array.isArray(v) && v.every((r) => {
    const x = r as any;
    const impactOk =
      typeof x.estimatedImpact === 'string' ||
      (!!x.estimatedImpact && typeof x.estimatedImpact === 'object' && typeof x.estimatedImpact.range === 'string');
    return !!x && typeof x === 'object'
      && typeof x.title === 'string'
      && typeof x.rationale === 'string'
      && FEATURES.includes(x.bloomreachFeature)
      && Array.isArray(x.steps)
      && impactOk
      && CONFIDENCE.includes(x.confidence);
  });
}
```

The strongest of the three. Two enum checks (`bloomreachFeature`, `confidence`), plus a union type for `estimatedImpact` (legacy string OR the newer `{ range, ... }` shape). The comment on line 41-42 is the interesting one: the model doesn't emit `id` — the system assigns it after validation. That's why the guard is `Omit<Recommendation, 'id'>[]` and downstream code adds `id: crypto.randomUUID()` before storing.

### Move 3 — the principle

Parse-don't-validate is the pattern any codebase with foreign-produced structured data should reach for. The generalization: type guards let you draw a bright line in the code between "unchecked input" and "typed value," and once you've drawn it, downstream code doesn't have to re-check anything. The type system carries the invariant forward.

For an AI-shaped codebase specifically: the model is the world's least-typed API. It responds with prose that *usually* contains JSON that *usually* matches your schema. Anything less than a full shape check at the boundary means the invariants leak downstream and every component becomes defensive. Draw the line once, at `validate.ts`.

## Elaborate

**Why not Zod?** Zod would give you a schema DSL and better error messages. It would also add a runtime dep and a schema-to-type sync problem (the schema drives the type; the types in `lib/mcp/types.ts` currently drive the guards). The hand-written guards are ~50 lines total across three functions. For an audit-shaped answer: this is fine at the current size; if the surface grows beyond ~5 shapes or the field lists get nested, Zod pays for itself.

**Why the fence-unwrap?** Because Anthropic's models — like every LLM — emit fenced JSON when asked for structured output. The exact string is model-specific and changes with model versions. The regex is deliberately permissive: optional `json` language tag, non-greedy content match, case-insensitive. If a future model switches to fenced with a different tag ("```yaml`), the regex still matches — you'd get invalid JSON at parse time and the substring fallback kicks in.

**Where the pattern originated.** Type predicates (`x is T`) landed in TypeScript 1.6 (2015). The parse-don't-validate framing is Alexis King's 2019 blog post but the underlying idea — trust boundaries where invariants get *established*, not just checked — goes back to Lampson and Saltzer.

**Adjacent concept in this repo:** `parseLiveMode` in `lib/data-source/index.ts` does the same job for the `mode` query param (validates a string against a `LiveMode` union). Different domain, same shape.

## Interview defense

### Q: "Why guards instead of just casting the parsed JSON?"

**Answer:** Because casting is a *lie*. `JSON.parse(text) as Anomaly[]` tells the compiler you know the shape; at runtime you know nothing. The first time the model emits `severity: "urgent"` — not one of the four values the UI supports — every component that switches on severity breaks in ways that are hard to attribute back to the model. The guard makes the failure loud at the boundary: return false, log it (in a real prod build), fall back to an empty result. The UI sees "no anomalies this run" instead of a runtime crash.

The trust axis version: casting keeps the input in the trusted-value bucket without proving it belongs there. The guard is the proof.

**Diagram to sketch:**

```
  Cast (broken):                          Guard (correct):
  ─────────────                           ────────────────
  unknown ──as X──► X                    unknown ──guard──► X | false
   │                                      │
   │ downstream trusts the shape          │ downstream trusts the shape
   │ that was never checked               │ that was verified
   ▼                                      ▼
  runtime crash                          clean fallback
  at the render layer
```

**Anchor:** `lib/mcp/validate.ts:17-27` — `isAnomalyArray` checks every field the UI reads.

### Q: "The isDiagnosis guard is weaker than the anomaly one. Why?"

**Answer:** Cost tradeoff. Diagnoses have deeply nested fields (`evidence[]` items each have `tool`, `result`; `hypothesesConsidered[]` items each have `hypothesis`, `supported`, `reasoning`). Checking every field in every array item would be a lot of code, and the UI for diagnoses (`components/investigation/EvidencePanel.tsx`) already renders defensively — items with missing pieces show placeholders. The guard checks the *outer* shape (conclusion is a string, evidence is an array, hypothesesConsidered is an array) and pushes the item-level validation to the render layer.

Honest answer: this is a tradeoff, not a strong choice. If a component crashes on a malformed evidence item in a real run, tighten the guard.

**Anchor:** `lib/mcp/validate.ts:29-35`.

### Q: "What would you have to change to move to Zod?"

**Answer:** Small change, big improvement in error messages. Write a `z.object({...})` schema per type in `lib/mcp/schema-zod.ts`, replace `isAnomalyArray(v)` with `AnomalyArraySchema.safeParse(v).success`, and expose the parse errors to the debug log. Two things you'd have to think about: (1) sync between `types.ts` and the schemas (Zod can derive types via `z.infer`, so you'd flip the source of truth); (2) bundle size — Zod adds ~15KB gzipped, which is fine for a server-side check.

At current scale (three shapes, ~50 lines of guards), the hand-written version is a wash. At double the surface, Zod wins.

**Anchor:** `lib/mcp/validate.ts` — 60 lines total.
